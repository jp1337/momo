# lib/

## Purpose
All server-side business logic and infrastructure. API routes import from here — no business logic lives in routes or components.

## Contents
- `auth.ts` — Auth.js v5 config: providers (GitHub, Discord, Google, optional OIDC), Drizzle adapter, session strategy
- `totp.ts` — Two-factor authentication (TOTP) business logic: generateTotpSetup (secret + QR data URL), verifyTotpCode, enableTotpForUser (verifies first code, encrypts secret, generates 10 backup codes — atomic), disableTotpForUser, regenerateBackupCodes, verifyUserTotpCode, consumeBackupCode (constant-time, single-use), getUserTotpStatus, **userHasSecondFactor** (method-agnostic gate — returns true when the user has TOTP *or* at least one registered passkey; single touchpoint used by every enforcement path), setup-cookie helpers (signSetupToken/verifySetupToken — HMAC-SHA256 with AUTH_SECRET, 10-min TTL, never persisted), session-token helpers (markSessionSecondFactorVerified, isSessionSecondFactorVerified, readSessionTokenFromCookieStore)
- `webauthn.ts` — Passkey (WebAuthn) business logic. Uses `@simplewebauthn/server` directly (no Auth.js Passkey provider, because it would require JWT sessions — Momo keeps DB sessions for revocation). Exports: getRpConfig (reads `WEBAUTHN_RP_ID`/`WEBAUTHN_RP_NAME`, defaults RP ID to the hostname of `NEXT_PUBLIC_APP_URL`), createRegistrationOptions, verifyRegistration (persists to `authenticators`), createDiscoverableLoginOptions (passwordless primary login — empty allowCredentials), createLoginOptionsForUser (2FA flow — allow-list scoped to a user), verifyLogin (looks up credential, verifies, updates counter + lastUsedAt), createPasskeyLoginSession (inserts a fresh `sessions` row with `second_factor_verified_at = now()` because passkeys are inherently MFA), listUserPasskeys / renamePasskey / deletePasskey for the settings UI, and signChallengeToken/verifyChallengeToken (HMAC-SHA256 over `AUTH_SECRET`, 5-min TTL, purpose-tagged `reg`/`login`/`sf` to prevent cross-flow replay — same pattern as `totp.ts::signSetupToken`)
- `env.ts` — Zod-validated env wrapper. **All** env var access must go through `serverEnv` or `clientEnv` exports here
- `db/index.ts` — Drizzle client (postgres driver), singleton pattern
- `db/schema.ts` — All table definitions: users, topics, tasks, task_completions, wishlist_items, achievements, user_achievements, authenticators (WebAuthn/Passkey credentials) + Auth.js tables (accounts, sessions, verification_tokens)
- `validators/passkey.ts` — Zod schemas for the `/api/auth/passkey/*` request bodies (RegistrationResponseJSON + AuthenticationResponseJSON shapes + rename body). Kept in its own file so the bulky WebAuthn types do not pollute `validators/index.ts`
- `tasks.ts` — getUserTasks, getTaskById, createTask (auto-assigns sortOrder within topic), updateTask, deleteTask, completeTask (coin award + streak + achievements, timezone-aware), uncompleteTask, breakdownTask (assigns sequential sortOrder), snoozeTask (hides task until date, clears isDailyQuest if active), unsnoozeTask, reorderTasks (bulk-updates sortOrder in transaction), **bulkUpdateTasks** (single-transaction bulk action: delete / complete / changeTopic / setPriority on up to 100 tasks; bulk-complete skips gamification and ignores recurring tasks)
- `topic-icons.ts` — TOPIC_ICONS map of 47 curated FA solid icons + resolveTopicIcon(key) with faFolder fallback
- `topics.ts` — getUserTopics (with task counts), getTopicById (with tasks, ordered by sortOrder ASC), createTopic, updateTopic, deleteTopic (reassigns tasks to null)
- `calendar.ts` — iCal calendar feed logic. `generateCalendarToken()` (256-bit, `momo_cal_<base64url>` + SHA-256 hash, mirrors `api-keys.ts` pattern), `createOrRotateCalendarToken()`/`revokeCalendarToken()` (write `users.calendar_feed_token_hash` + `calendar_feed_token_created_at`), `getCalendarFeedStatus()` (active + createdAt, never returns the hash), `getUserByCalendarToken()` (SHA-256 lookup — returns null for unknown/revoked), `buildIcsForUser(userId, baseUrl)` (pulls non-completed tasks with `due_date`/`next_due_date`, emits all-day VEVENTs via `ical-generator@10.1.0` — RECURRING tasks get `RRULE:FREQ=DAILY;INTERVAL=<recurrenceInterval>`, UID `task-<id>@momo` stable across polls, snoozed + sequential-blocked tasks included on purpose)
- `onboarding.ts` — Onboarding wizard business logic. `markOnboardingCompleted(userId)` sets `users.onboarding_completed = true`, `isOnboardingCompleted(userId)` checks the flag (defaults true for safety). Used by the `(app)` layout gate and the `/onboarding` page guard
- `templates.ts` — Curated topic templates (moving/taxes/fitness) as TypeScript constants. `TEMPLATES` catalogue, `getTemplate(key)`, `importTopicFromTemplate(userId, key, locale)` which resolves i18n strings via `getTranslations()` and inserts topic + tasks + `users.totalTasksCreated` increment in a single Drizzle transaction (mirrors `breakdownTask` pattern). Templates are code, not user data — adding one is a new entry in `TEMPLATES` plus a `templates.<key>.*` block in `messages/{de,en,fr}.json`
- `validators/index.ts` — Zod schemas: CreateTaskInput, UpdateTaskInput, SnoozeTaskInput, EnergyCheckinInput, ReorderTasksInput, **BulkTaskActionInput** (discriminated union: delete/complete/changeTopic/setPriority), CreateTopicInput, UpdateTopicInput, CreateWishlistItemInput, UpdateWishlistItemInput, UpdateProfileInput
- `daily-quest.ts` — selectDailyQuest, getCurrentDailyQuest, postponeDailyQuest (timezone-aware), forceSelectDailyQuest, **reselectQuestForEnergy** (idempotent re-roll triggered by energy check-in — only swaps when the current quest energy mismatches AND a strictly-better candidate exists; no-op for untagged/matching/completed quests), **pinTaskAsDailyQuest** (Undo for the energy swap). Energy-aware: pickBestTask prefers tasks matching user's daily energy check-in (soft preference, never blocks selection)
- `energy.ts` — Energy check-in business logic. recordEnergyCheckin (writes both `users.energy_level` cache + appends `energy_checkins` history row in one tx), getEnergyHistory (last N days, collapsed to one entry per local date — keeps the latest if the user re-checked-in), getEnergyLevelCounts (per-level day counts for the Stats weekly block), getEnergyCheckinDayCount (distinct days ever — drives the Stats empty-state hint). Multiple inserts per day are intentional — re-check-ins are preserved for future time-of-day analyses
- `gamification.ts` — updateStreak (timezone-aware), getLevelForCoins, checkAndUnlockAchievements, seedAchievements, getUserStats
- `habits.ts` — Habit-tracker read path for the `/habits` page. `getHabitsWithHistory(userId, year, timezone?)` returns every recurring task + its completion history for the requested year + rolling 30-day/7-day totals + a per-habit streak (current + all-time best, counted in periods of `recurrenceInterval ?? 1` days). `getEarliestCompletion` bounds the year selector. `computeHabitStreak(dates, interval, today)` is the pure streak algorithm — rolling period windows, grace for the currently-running period, multi-completion-in-one-period collapses to a single hit. Snoozed tasks never affect the streak (snooze pauses the reminder, not the record). Known caveat: for long intervals (monthly/yearly) the rolling window drifts against the calendar — the roadmap's "Erweiterte Wiederholungsregeln" item will fix this at source. All date arithmetic timezone-aware via `toLocalDateString` (mirrors `lib/date-utils.ts`)
- `date-utils.ts` — getLocalDateString, getLocalTomorrowString, getLocalYesterdayString — all timezone-aware via Intl.DateTimeFormat("en-CA")
- `push.ts` — savePushSubscription, sendPushNotification, sendStreakReminder, sendWeeklyReviewNotifications, **sendMorningBriefingNotifications** (daily digest: quest + due tasks + streak + recent achievements in one message; suppresses individual daily-quest and due-today jobs for digest users), **sendRecurringDueNotifications** (per-task individual notifications for recurring tasks due today; ≤3 tasks get individual notifications, >3 get bundled; suppressed when morning briefing is enabled) — Web Push via VAPID + multi-channel fan-out via lib/notifications.ts
- `notifications.ts` — Multi-channel notification system: NotificationChannel interface, NtfyChannel + PushoverChannel + TelegramChannel + EmailChannel classes, createChannel registry, sendToAllChannels fan-out (logs every attempt to notification_log), sendTestNotification, isEmailChannelAvailable (server-side gate based on SMTP_HOST/SMTP_FROM)
- `notification-log.ts` — Fire-and-forget notification delivery logging. `logNotification()` records each channel attempt to `notification_log` (never blocks delivery), `cleanupNotificationLog()` prunes entries older than 30 days (registered as daily cron job)
- `email-templates.ts` — renderEmailTemplate(payload, appUrl): newsletter-style HTML template (table-based, Outlook-compatible, Lora-heading, Waldgrün-accent, CTA button, footer with settings link); used by EmailChannel.send. Pure function, no side effects
- `vacation.ts` — Vacation mode business logic. `getVacationStatus(userId)` returns active status + end date, `activateVacationMode(userId, endDate, tz)` sets `users.vacationEndDate` and marks all RECURRING tasks as paused (`pausedAt/pausedUntil`), clears `isDailyQuest` on affected tasks (single transaction), `deactivateVacationMode(userId, tz)` shifts `nextDueDate` forward by actual pause days and clears pause columns (single transaction), `autoEndVacations()` cron handler finds users with expired vacations and deactivates them
- `weekly-review.ts` — getWeeklyReview(userId, timezone) — weekly performance summary (completions, postponements, coins, streak, top topics)
- `cron.ts` — Unified cron dispatcher. CRON_JOBS registry + runAllJobs(). Add new periodic jobs here — no Docker/endpoint changes needed
- `rate-limit.ts` — In-memory rate limiter (sliding window) applied to mutation API routes
- `wishlist.ts` — getUserWishlistItems, createWishlistItem, updateWishlistItem, deleteWishlistItem, buyWishlistItem, discardWishlistItem
- `api-keys.ts` — generateApiKey (256-bit), createApiKey, listApiKeys, revokeApiKey, resolveApiKeyUser
- `api-auth.ts` — resolveApiUser() — Bearer Token + Session Cookie, readonlyKeyResponse(); resolveVerifiedApiUser() (opt-in 2FA-aware variant — Bearer tokens are exempt; cookie sessions return TOTP_REQUIRED / TOTP_SETUP_REQUIRED if not verified) + verifiedAuthErrorResponse()
- `openapi.ts` — Full OpenAPI 3.1.0 specification object (served at /api/openapi.json)
- `statistics.ts` — getUserStatistics(userId), getAdminStatistics() — aggregated stats for /stats and /admin pages
- `export.ts` — buildUserExport(userId) — GDPR data export (all user data as JSON)
- `users.ts` — deleteUser(userId) — full account deletion cascade; updateUserProfile(userId, data) — update name/email/avatar; processProfileImage(dataUrl) — resize to 200×200 WebP via Sharp
- `utils/crypto.ts` — Cryptographic helpers: timingSafeEqual (constant-time string compare), encryptSecret/decryptSecret (AES-256-GCM with TOTP_ENCRYPTION_KEY, fresh IV per call, AuthTag separate), hashBackupCode (SHA-256 hex; mirrors api-keys pattern)
- `client/coin-events.ts` — Client-safe module: `COINS_EARNED_EVENT` const + `dispatchCoinsEarned(delta)` with NaN/Infinity guard. Shared by TaskList, TopicDetailView, DailyQuestCard and CoinCounter.

## Patterns
- Business logic functions go directly in `lib/` (e.g. `lib/tasks.ts`, `lib/daily-quest.ts`)
- Every exported function gets a JSDoc comment
- DB queries use Drizzle ORM — no raw SQL strings
- Always filter by `userId` from session to scope data per user
