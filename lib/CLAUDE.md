# lib/

## Purpose
All server-side business logic and infrastructure. API routes import from here ‚Äî no business logic lives in routes or components.

## Contents
- `auth.ts` ‚Äî Auth.js v5 config: providers (GitHub, Discord, Google, optional OIDC), Drizzle adapter, session strategy
- `totp.ts` ‚Äî Two-factor authentication (TOTP) business logic: generateTotpSetup (secret + QR data URL), verifyTotpCode, enableTotpForUser (verifies first code, encrypts secret, generates 10 backup codes ‚Äî atomic), disableTotpForUser, regenerateBackupCodes, verifyUserTotpCode, consumeBackupCode (constant-time, single-use), getUserTotpStatus, **userHasSecondFactor** (method-agnostic gate ‚Äî returns true when the user has TOTP *or* at least one registered passkey; single touchpoint used by every enforcement path), setup-cookie helpers (signSetupToken/verifySetupToken ‚Äî HMAC-SHA256 with AUTH_SECRET, 10-min TTL, never persisted), session-token helpers (markSessionSecondFactorVerified, isSessionSecondFactorVerified, readSessionTokenFromCookieStore)
- `webauthn.ts` ‚Äî Passkey (WebAuthn) business logic. Uses `@simplewebauthn/server` directly (no Auth.js Passkey provider, because it would require JWT sessions ‚Äî Momo keeps DB sessions for revocation). Exports: getRpConfig (reads `WEBAUTHN_RP_ID`/`WEBAUTHN_RP_NAME`, defaults RP ID to the hostname of `NEXT_PUBLIC_APP_URL`), createRegistrationOptions, verifyRegistration (persists to `authenticators`), createDiscoverableLoginOptions (passwordless primary login ‚Äî empty allowCredentials), createLoginOptionsForUser (2FA flow ‚Äî allow-list scoped to a user), verifyLogin (looks up credential, verifies, updates counter + lastUsedAt), createPasskeyLoginSession (inserts a fresh `sessions` row with `second_factor_verified_at = now()` because passkeys are inherently MFA), listUserPasskeys / renamePasskey / deletePasskey for the settings UI, and signChallengeToken/verifyChallengeToken (HMAC-SHA256 over `AUTH_SECRET`, 5-min TTL, purpose-tagged `reg`/`login`/`sf` to prevent cross-flow replay ‚Äî same pattern as `totp.ts::signSetupToken`)
- `env.ts` ‚Äî Zod-validated env wrapper. **All** env var access must go through `serverEnv` or `clientEnv` exports here
- `db/index.ts` ‚Äî Drizzle client (postgres driver), singleton pattern
- `db/schema.ts` ‚Äî All table definitions: users, topics, tasks, task_completions, wishlist_items, achievements, user_achievements, authenticators (WebAuthn/Passkey credentials) + Auth.js tables (accounts, sessions, verification_tokens)
- `validators/passkey.ts` ‚Äî Zod schemas for the `/api/auth/passkey/*` request bodies (RegistrationResponseJSON + AuthenticationResponseJSON shapes + rename body). Kept in its own file so the bulky WebAuthn types do not pollute `validators/index.ts`
- `tasks.ts` ‚Äî getUserTasks, getTaskById, createTask (auto-assigns sortOrder within topic), updateTask, deleteTask, completeTask (coin award + streak + achievements, timezone-aware), uncompleteTask, breakdownTask (assigns sequential sortOrder), snoozeTask (hides task until date, clears isDailyQuest if active), unsnoozeTask, reorderTasks (bulk-updates sortOrder in transaction)
- `topic-icons.ts` ‚Äî TOPIC_ICONS map of 47 curated FA solid icons + resolveTopicIcon(key) with faFolder fallback
- `topics.ts` ‚Äî getUserTopics (with task counts), getTopicById (with tasks, ordered by sortOrder ASC), createTopic, updateTopic, deleteTopic (reassigns tasks to null)
- `calendar.ts` ‚Äî iCal calendar feed logic. `generateCalendarToken()` (256-bit, `momo_cal_<base64url>` + SHA-256 hash, mirrors `api-keys.ts` pattern), `createOrRotateCalendarToken()`/`revokeCalendarToken()` (write `users.calendar_feed_token_hash` + `calendar_feed_token_created_at`), `getCalendarFeedStatus()` (active + createdAt, never returns the hash), `getUserByCalendarToken()` (SHA-256 lookup ‚Äî returns null for unknown/revoked), `buildIcsForUser(userId, baseUrl)` (pulls non-completed tasks with `due_date`/`next_due_date`, emits all-day VEVENTs via `ical-generator@10.1.0` ‚Äî RECURRING tasks get `RRULE:FREQ=DAILY;INTERVAL=<recurrenceInterval>`, UID `task-<id>@momo` stable across polls, snoozed + sequential-blocked tasks included on purpose)
- `templates.ts` ‚Äî Curated topic templates (moving/taxes/fitness) as TypeScript constants. `TEMPLATES` catalogue, `getTemplate(key)`, `importTopicFromTemplate(userId, key, locale)` which resolves i18n strings via `getTranslations()` and inserts topic + tasks + `users.totalTasksCreated` increment in a single Drizzle transaction (mirrors `breakdownTask` pattern). Templates are code, not user data ‚Äî adding one is a new entry in `TEMPLATES` plus a `templates.<key>.*` block in `messages/{de,en,fr}.json`
- `validators/index.ts` ‚Äî Zod schemas: CreateTaskInput, UpdateTaskInput, SnoozeTaskInput, EnergyCheckinInput, ReorderTasksInput, CreateTopicInput, UpdateTopicInput, CreateWishlistItemInput, UpdateWishlistItemInput, UpdateProfileInput
- `daily-quest.ts` ‚Äî selectDailyQuest, getCurrentDailyQuest, postponeDailyQuest (timezone-aware), forceSelectDailyQuest, **reselectQuestForEnergy** (idempotent re-roll triggered by energy check-in ‚Äî only swaps when the current quest energy mismatches AND a strictly-better candidate exists; no-op for untagged/matching/completed quests), **pinTaskAsDailyQuest** (Undo for the energy swap). Energy-aware: pickBestTask prefers tasks matching user's daily energy check-in (soft preference, never blocks selection)
- `energy.ts` ‚Äî Energy check-in business logic. recordEnergyCheckin (writes both `users.energy_level` cache + appends `energy_checkins` history row in one tx), getEnergyHistory (last N days, collapsed to one entry per local date ‚Äî keeps the latest if the user re-checked-in), getEnergyLevelCounts (per-level day counts for the Stats weekly block), getEnergyCheckinDayCount (distinct days ever ‚Äî drives the Stats empty-state hint). Multiple inserts per day are intentional ‚Äî re-check-ins are preserved for future time-of-day analyses
- `gamification.ts` ‚Äî updateStreak (timezone-aware), getLevelForCoins, checkAndUnlockAchievements, seedAchievements, getUserStats
- `date-utils.ts` ‚Äî getLocalDateString, getLocalTomorrowString, getLocalYesterdayString ‚Äî all timezone-aware via Intl.DateTimeFormat("en-CA")
- `push.ts` ‚Äî savePushSubscription, sendPushNotification, sendStreakReminder, sendWeeklyReviewNotifications ‚Äî Web Push via VAPID + multi-channel fan-out via lib/notifications.ts
- `notifications.ts` ‚Äî Multi-channel notification system: NotificationChannel interface, NtfyChannel + PushoverChannel + TelegramChannel + EmailChannel classes, createChannel registry, sendToAllChannels fan-out, sendTestNotification, isEmailChannelAvailable (server-side gate based on SMTP_HOST/SMTP_FROM)
- `email-templates.ts` ‚Äî renderEmailTemplate(payload, appUrl): newsletter-style HTML template (table-based, Outlook-compatible, Lora-heading, Waldgr√ºn-accent, CTA button, footer with settings link); used by EmailChannel.send. Pure function, no side effects
- `weekly-review.ts` ‚Äî getWeeklyReview(userId, timezone) ‚Äî weekly performance summary (completions, postponements, coins, streak, top topics)
- `cron.ts` ‚Äî Unified cron dispatcher. CRON_JOBS registry + runAllJobs(). Add new periodic jobs here ‚Äî no Docker/endpoint changes needed
- `rate-limit.ts` ‚Äî In-memory rate limiter (sliding window) applied to mutation API routes
- `wishlist.ts` ‚Äî getUserWishlistItems, createWishlistItem, updateWishlistItem, deleteWishlistItem, buyWishlistItem, discardWishlistItem
- `api-keys.ts` ‚Äî generateApiKey (256-bit), createApiKey, listApiKeys, revokeApiKey, resolveApiKeyUser
- `api-auth.ts` ‚Äî resolveApiUser() ‚Äî Bearer Token + Session Cookie, readonlyKeyResponse(); resolveVerifiedApiUser() (opt-in 2FA-aware variant ‚Äî Bearer tokens are exempt; cookie sessions return TOTP_REQUIRED / TOTP_SETUP_REQUIRED if not verified) + verifiedAuthErrorResponse()
- `openapi.ts` ‚Äî Full OpenAPI 3.1.0 specification object (served at /api/openapi.json)
- `statistics.ts` ‚Äî getUserStatistics(userId), getAdminStatistics() ‚Äî aggregated stats for /stats and /admin pages
- `export.ts` ‚Äî buildUserExport(userId) ‚Äî GDPR data export (all user data as JSON)
- `users.ts` ‚Äî deleteUser(userId) ‚Äî full account deletion cascade; updateUserProfile(userId, data) ‚Äî update name/email/avatar; processProfileImage(dataUrl) ‚Äî resize to 200√ó200 WebP via Sharp
- `utils/crypto.ts` ‚Äî Cryptographic helpers: timingSafeEqual (constant-time string compare), encryptSecret/decryptSecret (AES-256-GCM with TOTP_ENCRYPTION_KEY, fresh IV per call, AuthTag separate), hashBackupCode (SHA-256 hex; mirrors api-keys pattern)
- `client/coin-events.ts` ‚Äî Client-safe module: `COINS_EARNED_EVENT` const + `dispatchCoinsEarned(delta)` with NaN/Infinity guard. Shared by TaskList, TopicDetailView, DailyQuestCard and CoinCounter.

## Patterns
- Business logic functions go directly in `lib/` (e.g. `lib/tasks.ts`, `lib/daily-quest.ts`)
- Every exported function gets a JSDoc comment
- DB queries use Drizzle ORM ‚Äî no raw SQL strings
- Always filter by `userId` from session to scope data per user
