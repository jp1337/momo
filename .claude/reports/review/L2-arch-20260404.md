**Date:** 2026-04-04
**Reviewer:** Claude Sonnet 4.6 (system mode)
**Branch:** main

## Executive Summary

The Phase 11 session introduced a coherent timezone-awareness layer (`lib/date-utils.ts`, `getLocalDateString` family) and threaded it correctly through the completion and postpone paths. The work is architecturally sound in principle. However, **the threading is incomplete**: the daily quest selection algorithm (`selectDailyQuest`, `pickBestTask`, `forceSelectDailyQuest`) was not updated and still computes the date boundary using a private UTC-only helper. This creates a split-brain condition where the same user action (completing vs. selecting a quest) uses different date logic. The split-brain is identified as a BLOCKING issue in the L1 peer review and confirmed by direct code inspection here.

The remaining five criteria — coin event pattern, migration script, date-utils design, dependency direction, and shared hook extraction — all have architectural observations documented below.

## 1. Timezone Threading Consistency

### Call Chain Map

The following table traces timezone from client browser through to the database write for each affected operation.

| Operation | Client sends timezone | API route validates | lib function signature | lib uses getLocalDateString | Verdict |
|---|---|---|---|---|---|
| Complete task | Yes — `task-list.tsx:261`, `topic-detail-view.tsx:76`, `daily-quest-card.tsx:128` | No (manual `typeof` check, no length cap) | `completeTask(id, userId, timezone?)` | Yes — passed to `updateStreak` | BLOCKING gap at route (no max(64)) |
| Postpone quest | Yes — `daily-quest-card.tsx:185` | Yes — Zod `z.string().max(64)` in `PostponeBodySchema` | `postponeDailyQuest(taskId, userId, timezone?)` | Yes — `getLocalDateString` + `getLocalTomorrowString` | Correct |
| Select quest (daily) | Not sent | N/A | `selectDailyQuest(userId)` — no timezone param | No — uses private `getTodayString()` (UTC) | BLOCKING gap in lib |
| Select quest (force) | Not sent | N/A | `forceSelectDailyQuest(userId)` — no timezone param | No — uses private `getTodayString()` (UTC) | BLOCKING gap in lib |
| Update streak | Not sent independently; received via `completeTask` | N/A | `updateStreak(userId, tx, timezone?)` | Yes | Correct |

### The Split-Brain Problem

`daily-quest.ts` at line 43–49 still contains a private `getTodayString()` function that is structurally identical to the server-UTC fallback branch of `getLocalDateString()`. Three functions call it:

- `pickBestTask` (line 149) — determines which task is selected as the daily quest
- `selectDailyQuest` (line 247) — wraps `pickBestTask` and checks "completed today"
- `forceSelectDailyQuest` (line 335) — duplicate of the algorithm outside a transaction

For a user in UTC+9, between midnight local time and 09:00 UTC, the server date reads as "yesterday". All three functions will compare task due dates against yesterday's date string, causing overdue tasks from two days ago to still appear as "one day overdue" and tasks that became due at local midnight to not yet appear as due. The `postponeDailyQuest` function, which runs after selection, correctly uses the user's local date. This means the quest can be postponed to "tomorrow (local)" while having been selected using "yesterday (UTC)" as the cutoff — a logically inconsistent boundary.

### Root Cause

The partial update in this session updated `postponeDailyQuest` and `completeTask` but did not propagate `timezone` into the selection path. The selection path is invoked by `GET /api/daily-quest/route.ts` which does not accept or forward a timezone. The missing threading requires:

1. `selectDailyQuest` and `forceSelectDailyQuest` to accept `timezone?: string | null`
2. `pickBestTask` to receive and use the timezone parameter
3. `GET /api/daily-quest/route.ts` to read the timezone from the request (query param or header) and pass it through
4. The `DailyQuestCard` parent (the dashboard Server Component) to forward the timezone, or the client component to supply it via a separate API parameter

Until this is done, `getTodayString()` should at minimum be replaced with `getLocalDateString()` (no argument — the fallback is identical and removes the code duplication risk identified in L1 Finding 12).

### Recurring Task `nextDueDate` (Non-Blocking)

`lib/tasks.ts` lines 259–262 compute `nextDueDate` via `new Date(now).toISOString().split("T")[0]`. This extracts the UTC date of the completion moment. For users in eastern UTC+ zones this produces a date one calendar day behind their local date. The fix follows the same pattern: use `getLocalDateString(timezone)` as the base date and add `intervalDays` calendar days to it. Since `completeTask` already receives `timezone`, the data is present; it is not yet used for this calculation.

## 2. Cross-Component Coin Event Pattern

### Current Pattern

Three client components dispatch the `coinsEarned` CustomEvent on `window`:

| Component | Dispatch location | Event detail |
|---|---|---|
| `task-list.tsx` | `handleComplete` (line 275), `handleUncomplete` (line 304) | `{ delta: +N }` / `{ delta: -N }` |
| `topic-detail-view.tsx` | `handleComplete` (line 92), `handleUncomplete` (line 118) | `{ delta: +N }` / `{ delta: -N }` |
| `daily-quest-card.tsx` | `handleComplete` (line 148) | `{ delta: +N }` |

`layout/coin-counter.tsx` listens for this event and updates the displayed balance without a full page reload.

### Assessment

The pattern is sound for this application size. `window.dispatchEvent` with a named CustomEvent is a legitimate client-side pub/sub mechanism in React applications where a global store (Redux, Zustand) would be overkill. There are three specific properties worth documenting architecturally:

**Why this pattern is acceptable here:**

- The CoinCounter lives in the navbar, which is outside the React subtree of all three components. A shared prop or context passed through the layout would require lifting state to the root layout and re-rendering the entire authenticated shell on every coin change. The CustomEvent avoids this.
- The delta model (`+N` for award, `-N` for refund) is simpler than re-fetching the absolute balance, and the source of truth (server DB) is always authoritative — a stale delta only causes a transient display discrepancy, not data loss.
- The pattern is already used consistently across all three call sites with the same event name and detail shape.

**Where the pattern creates risk:**

- There is no central definition of the event name `"coinsEarned"` or the detail type `{ delta: number }`. Each component hard-codes both. If the event name or shape is changed in one place, the other two will silently drift.
- The `DailyQuestCard` does not handle the uncomplete case (there is no undo-complete for the daily quest). This is correct behavior, but the asymmetry means a developer adding an undo button to the quest card needs to know to dispatch the negative delta event — there is no single place documenting this contract.

**Architectural recommendation (non-blocking):**

Define a single typed helper in a shared module — for example `lib/client/coin-events.ts` (client-only, `"use client"` file) — that exports:

```
dispatchCoinsEarned(delta: number): void
dispatchCoinsRefunded(coinValue: number): void
```

and the event name constant. All three components import from this file. This consolidates the contract without changing the event mechanism. The cost is one new file; the benefit is a single place to audit and update the pattern.

This is a maintenance concern, not a correctness concern. The current three-way duplication works correctly.

## 3. Migration Script Evolution

### Current Object Detection Coverage

`scripts/migrate.mjs` now handles three types of DDL object detection in `isMigrationAppliedInDb`:

| Object type | Parser function | DB inspector | Matched pattern |
|---|---|---|---|
| Tables | `parseCreatedTables` | `tableExists` (information_schema.tables) | `CREATE TABLE [IF NOT EXISTS] "name"` |
| Indexes | `parseCreatedIndexes` | `indexExists` (pg_indexes) | `CREATE [UNIQUE] INDEX "name"` |
| Columns | `parseAlterAddColumns` | `columnExists` (information_schema.columns) | `ALTER TABLE "table" ADD COLUMN [IF NOT EXISTS] "column"` |

### Pattern Extensibility

The three-parser pattern is extensible. Each object type follows the same contract: a pure function that parses SQL into a list of identifiers, and an async function that queries the DB. Adding a fourth type (e.g., ENUM type detection, view detection) requires adding one parser function and one inspector function and registering them in `isMigrationAppliedInDb`. The shape is consistent.

### Blind Spots

**Identifier regex coverage (L1 Finding 5 confirmed):**

All three parsers use `\w+` in the capturing group. PostgreSQL allows quoted identifiers containing hyphens, spaces, dots, and unicode characters. The Drizzle schema for this project uses snake_case throughout, so the current schema is safe. The risk is latent: a future migration with a hyphenated or schema-prefixed identifier will fail silently — `parseAlterAddColumns` will return an empty array, `isMigrationAppliedInDb` will return `false`, and the safe fallback will be to let `migrate()` attempt the migration (which may fail with "column already exists" if the schema was previously applied manually). This is the same failure mode the script was designed to prevent.

**Destructive DDL not covered:**

`DROP COLUMN`, `DROP TABLE`, `RENAME COLUMN`, `ALTER COLUMN TYPE` are not parsed. A migration that drops a column would pass `tableExists` (the table still exists) and the column check is not applicable, so `isMigrationAppliedInDb` would return `true` for a `CREATE TABLE` migration that was followed by a `DROP COLUMN` in a later migration. This is not a current problem because Drizzle migrations are append-only in this project, but it is a blind spot for any future destructive migration.

**ENUM type detection not present:**

The problem statement in the script's leading comment mentions "partial DDL (e.g. ENUM types)" as a known failure mode. ENUM types created with `CREATE TYPE ... AS ENUM` are not detected by any of the three parsers. If a migration creates both an ENUM type and a table (common in Drizzle-generated migrations), `parseCreatedTables` will find the table, `tableExists` will return true, and the migration will be seeded — even if the ENUM creation failed on a previous run and the ENUM does not exist. This is an existing gap from before this session. The fix would be a fourth parser for `CREATE TYPE "name" AS ENUM` paired with a `pg_type` inspector.

**Verdict:**

The migration script's three-type detection pattern is well-structured and extensible. The identifier regex limitation is the most actionable near-term gap. The ENUM blind spot is a pre-existing issue that should be addressed before any migration that creates an ENUM type is added.

## 4. `lib/date-utils.ts` Design

### Module Placement

`lib/date-utils.ts` is placed in `lib/` alongside the business logic modules. This is correct. It is a pure utility module with no DB dependency and no framework dependency — it could theoretically live in `lib/utils/` but the current flat placement is consistent with how `lib/rate-limit.ts` and `lib/env.ts` are organized. No change recommended.

### Function Signature Consistency

All three exported functions follow the same signature shape:

```typescript
function getLocal*String(timezone?: string | null): string
```

This is consistent. The `timezone` parameter is optional and nullable in all three, which allows callers to pass the raw value from a parsed request body without needing to handle the undefined case at the call site. The `Intl.DateTimeFormat` constructor accepts `undefined` as a timezone (uses system default), but the code explicitly guards against this by testing `if (timezone)` before constructing the formatter — so `null` and `undefined` both fall through to the server-local fallback. This is correct and consistent.

### DST Edge Case in `getLocalTomorrowString` and `getLocalYesterdayString` (L1 Finding 10)

Both functions compute the offset date by calling `setDate(n ± 1)` on a `Date` object created with `new Date()` (server wall-clock), then pass the result to `Intl.DateTimeFormat` for the target timezone. The DST concern raised in the L1 review is technically present but the practical impact is limited: `setDate` operates on the local date representation of the `Date` object, and the subsequent `Intl.DateTimeFormat` call re-interprets the timestamp in the user's timezone. Across a DST spring-forward (23-hour day), `setDate(today + 1)` correctly advances to the next calendar day in UTC terms; the `Intl` formatter will then render the correct local date. The actual risk scenario would require a DST transition to cause the UTC midpoint of "tomorrow" to map to the wrong local date, which would need a DST shift of more than 12 hours — this does not exist in practice. The finding is valid as a theoretical concern but has no known real-world manifestation. The fallback path (server-local) is the code that actually has the DST risk for non-UTC servers; the `Intl` path is robust.

### Missing Unit Tests

The three functions have no test coverage. Given that they are the foundation of the entire timezone-awareness feature, and given that a regression here would silently produce wrong dates for all timezone-affected users, this is an architectural gap. The functions are pure (no DB, no side effects) and deterministic for a given input timestamp — they are ideal candidates for unit tests using fixed Date mocks.

## 5. Dependency Direction

### Rule

Per `CLAUDE.md` and the project architecture: components must not import from `lib/` directly. The intended dependency direction is component → API → lib.

### Violations Found

**`components/dashboard/daily-quest-card.tsx` imports `lib/topic-icons.ts`:**

```typescript
import { resolveTopicIcon } from "@/lib/topic-icons";
```

This is a direct lib import from a component. `lib/topic-icons.ts` is a pure client-safe utility — it contains no DB access, no server-only code, and no environment variable reads. It exports a map of FontAwesome icon definitions and a resolver function. The practical risk of this violation is zero. However, it is an architectural inconsistency.

The options are:

1. Move `lib/topic-icons.ts` to `lib/client/topic-icons.ts` or `utils/topic-icons.ts` to indicate it is client-safe
2. Accept the violation as a documented exception for pure utility modules with no server-side state

This same import pattern is used in `components/topics/topic-card.tsx` (not reviewed in this session but likely present). Establishing a `lib/client/` subdirectory for client-safe lib utilities would resolve the ambiguity without requiring a component-to-API roundtrip for static data.

**No other direct lib imports were found** in the three components reviewed (`task-list.tsx`, `topic-detail-view.tsx`, `daily-quest-card.tsx`). All data mutations go through `fetch()` API calls as required.

## 6. Pattern Consistency — Duplicate Complete/Uncomplete Logic

### Current Duplication

`TaskList` and `TopicDetailView` implement structurally identical `handleComplete` and `handleUncomplete` callbacks:

| Element | TaskList | TopicDetailView |
|---|---|---|
| Timezone capture | `Intl.DateTimeFormat().resolvedOptions().timeZone` | Identical |
| API call | `POST /api/tasks/${id}/complete` with `{timezone}` | Identical |
| Confetti trigger | `triggerSmallConfetti()` | Identical |
| Coin event dispatch | `window.dispatchEvent(new CustomEvent("coinsEarned", ...))` | Identical |
| Level-up state | `setLevelUp(data.newLevel)` | Identical |
| Achievement state | `setPendingAchievements` append | Identical |
| Refresh call | `refreshTasks()` (different URL) | Identical (different URL) |
| Error handling | Silent catch | Silent catch, identical |
| Uncomplete dispatch | `delta: -refunded` | Identical |

The only meaningful difference between the two implementations is the `refreshTasks` URL (`/api/tasks` vs. `/api/tasks?topicId=${topicId}`).

### Should This Be a Shared Hook?

Yes. The duplication is now significant enough to warrant extraction. The case for a shared hook:

- **Maintenance burden:** Any change to the complete/uncomplete flow (adding a new animation, changing the coin event shape, adding error toast feedback) must be made in two places and risks drift. This session already demonstrates the risk — both components were updated together, but a future developer may update one and miss the other.
- **Testing surface:** A single `useTaskCompletion(onRefresh: () => Promise<void>)` hook is trivially testable; duplicated component logic is not.
- **Established React pattern:** The logic is stateful (levelUp, pendingAchievements, isCompleting) and side-effectful (fetch + DOM events + animations) — exactly the kind of logic hooks are designed to encapsulate.

**Proposed hook signature:**

```typescript
// hooks/use-task-completion.ts
function useTaskCompletion(
  onRefresh: () => Promise<void>
): {
  handleComplete: (id: string) => Promise<void>;
  handleUncomplete: (id: string) => Promise<void>;
  levelUp: { level: number; title: string } | null;
  clearLevelUp: () => void;
  pendingAchievements: AchievementItem[];
  clearAchievements: () => void;
}
```

The `onRefresh` callback is the only point of divergence between the two components. Each component passes its own `refreshTasks` function. The hook manages the shared state (`levelUp`, `pendingAchievements`) and the shared side effects (fetch, event dispatch, confetti). The component renders the LevelUpOverlay and AchievementToast using the hook's state.

`DailyQuestCard` would also benefit from this hook but has a different completion path (it sets a local `isCompleted` state and does not refresh the task list). It could share a partial version of the hook or accept the divergence as intentional.

**Priority:** Non-blocking. The duplication works correctly. Extraction should be scheduled for the next refactoring session, not done alongside a bugfix session where the risk of regression is higher.

## 8. Decision Record: Why the Coin CustomEvent Pattern Was Not Changed

The L1 review identified three components dispatching `"coinsEarned"` CustomEvents. This review considered whether the pattern should be replaced with a React context or server-sent state update. The decision is to retain the CustomEvent pattern with the recommendation to centralize the event definition (A4 above).

**Rationale:**

The `CoinCounter` component in the navbar lives outside the React subtree of all three dispatch sites. Lifting coin state to the root layout would make the entire authenticated shell a client component or require a context provider wrapping the full layout — either adds complexity that exceeds the benefit for a single counter. The CustomEvent is a clean, intentional decoupling between the layout shell and the page components. The existing pattern is correct; it only lacks the central definition that would make the contract explicit.
