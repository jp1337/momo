### `lib/date-utils.ts`

**Overall:** Clean, well-documented, correct use of `Intl.DateTimeFormat("en-CA")` for YYYY-MM-DD output. The silent catch-and-fallback pattern for invalid timezone strings is correct and matches the intent.

#### Finding 10 — NON-BLOCKING: DST edge case in `getLocalTomorrowString`

**Lines 49–69**

`getLocalTomorrowString` and `getLocalYesterdayString` both compute the offset date by calling `setDate()` on a `Date` object in **server** local time and then pass the resulting timestamp to `Intl.DateTimeFormat` for the target timezone. This is correct in the common case but subtly wrong across a DST boundary: if the server is in UTC and the user is in a UTC+ timezone, calling `setDate(today.getDate() + 1)` advances the UTC timestamp by exactly 24 hours. When DST causes a 23- or 25-hour day in the user's timezone, the resulting local date may be wrong by one day.

The correct fix is to compute tomorrow as `today + 86_400_000 ms` (always exactly 24 hours of UTC wall time), which Intl.DateTimeFormat then formats in the user's timezone — but that still has the same DST ambiguity on the day the clock changes. The most robust pattern is to derive tomorrow by formatting `today`, parsing the YYYY-MM-DD, and adding one calendar day in the Intl API itself. This is an edge case affecting at most two days per year per DST transition, so it is NON-BLOCKING.

### `lib/daily-quest.ts`

#### Finding 2 — BLOCKING: Private `getTodayString()` ignores user timezone

**Lines 43–49, 149, 247, 335**

`daily-quest.ts` contains its own private `getTodayString()` that always returns the server's local date (UTC in containers). This function is used by `pickBestTask` (line 149), `selectDailyQuest` (line 247), and `forceSelectDailyQuest` (line 335). As a result, the overdue/due comparison for the daily quest algorithm uses the **server UTC date**, not the user's local date.

Compare `postponeDailyQuest`, which correctly calls `getLocalDateString(timezone)` (lines 445, 488). The selection logic has no such correction — a user in UTC+9 working after midnight local time will have yesterday's server date used for the priority algorithm, causing incorrectly-aged tasks to appear or not appear as the daily quest.

The private helper should be removed. `pickBestTask` and the free functions in `selectDailyQuest`/`forceSelectDailyQuest` need to accept a `timezone` parameter (or derive it from the user record if not available at call time). At minimum, the `getTodayString()` calls in `pickBestTask` and `selectDailyQuest` should be replaced with `getLocalDateString()` with no timezone argument (which already falls back to server local time), removing the code duplication. Full timezone support would require threading the timezone through `selectDailyQuest`.

#### Finding 7 — NON-BLOCKING: `todayStart` computed with local server time

**Line 248**

```typescript
const todayStart = new Date(`${today}T00:00:00`);
```

This produces midnight in the **server's local timezone**. In a UTC container this is correct. On any non-UTC server this is `today` at local midnight, not UTC midnight, creating an inconsistent comparison boundary with `tasks.completedAt` which is stored as a UTC timestamp. Should be:

```typescript
const todayStart = new Date(`${today}T00:00:00Z`); // UTC midnight
```

#### Finding 12 — NIT: `getTodayString()` private helper lacks JSDoc and duplicates `getLocalDateString` fallback

**Lines 43–49**

The private function has no JSDoc and is an exact copy of the fallback branch of `getLocalDateString`. If a developer later changes the fallback logic in `date-utils.ts`, this copy will drift silently.

### `lib/tasks.ts`

#### Finding 3 — NON-BLOCKING: RECURRING `nextDueDate` computed in server UTC

**Lines 259–262**

```typescript
const nextDue = new Date(now);
nextDue.setDate(nextDue.getDate() + intervalDays);
const nextDueStr = nextDue.toISOString().split("T")[0]; // YYYY-MM-DD
```

`now` is `new Date()` (UTC). `toISOString().split("T")[0]` extracts the UTC date. For a user in UTC+10 completing a recurring task at 23:30 their local time (13:30 UTC), `now` is already "yesterday" UTC, so `nextDueStr` will be off by one day relative to their local calendar. The timezone is available in the function signature — it should be used here via `getLocalDateString(timezone)` to anchor the calculation in the user's local day, then add `intervalDays` to that date string.

### `scripts/migrate.mjs`

#### Finding 5 — NON-BLOCKING: `parseAlterAddColumns` regex misses hyphenated/schema-qualified identifiers

**Line 94**

```javascript
const re = /ALTER TABLE\s+"?(\w+)"?\s+ADD COLUMN(?:\s+IF NOT EXISTS)?\s+"?(\w+)"?/gi;
```

`\w+` only matches `[A-Za-z0-9_]`. It will fail to capture table or column names containing hyphens (e.g. `"api-keys"`) because the regex consumes the opening `"` then tries to match `api` before stopping at `-`. However, since the project's Drizzle schema uses snake_case identifiers, this is unlikely to cause a production failure with the current schema. The risk is that a future migration with a hyphenated identifier will be silently treated as having no ALTER TABLE statements, causing `isMigrationAppliedInDb` to return `false` (safe fallback — migrate() will run it normally). Low practical risk.

Additionally, the function handles only `ADD COLUMN` but not `DROP COLUMN` or `ALTER COLUMN` — if a migration removes a column and `tableExists` returns true for its table, the migration could be incorrectly seeded as applied when it was not. This is an existing limitation acknowledged in the comment (`Cannot verify — let migrate() handle it` pattern) and is acceptable given the script's stated scope.

#### Finding 13 — NIT: `tableExists`, `indexExists`, `columnExists` lack JSDoc

**Lines 61–86**

All other functions in the file have JSDoc comments. These three utility functions are the exception.

### `app/api/tasks/[id]/complete/route.ts`

#### Finding 1 — BLOCKING: No server-side validation of the `timezone` string

**Lines 42–48**

```typescript
const body = await request.json() as { timezone?: string | null };
if (typeof body?.timezone === "string") timezone = body.timezone;
```

The timezone string is user-supplied and passed directly into `completeTask`, which passes it to `getLocalDateString` and `updateStreak`. The `Intl.DateTimeFormat` constructor will throw for invalid timezone strings (caught silently in `getLocalDateString`), so there is no remote code execution risk. However, there is no length cap on the timezone string. A client can submit an arbitrarily long string that will be processed before the try/catch catches the Intl error.

By contrast, `app/api/daily-quest/postpone/route.ts` correctly uses Zod with `.max(64)` for the timezone field. The complete route should apply the same validation. Recommended fix:

```typescript
const TimezoneSchema = z.string().max(64).optional().nullable();
// or inline:
if (typeof body?.timezone === "string" && body.timezone.length <= 64) {
  timezone = body.timezone;
}
```

This is BLOCKING because it is a security inconsistency between two routes that perform the same operation — one validates, one does not.

### `components/tasks/task-list.tsx`

#### Finding 4 — NON-BLOCKING: `recurrenceInterval` missing from `Task` interface, worked around with unsafe cast

**Lines 35–42 (interface), line 608**

The `Task` interface in `task-list.tsx` does not include `recurrenceInterval`. The edit form pre-fill at line 608 works around this with:

```typescript
recurrenceInterval: editingTask.type === "RECURRING"
  ? String((editingTask as unknown as { recurrenceInterval: number }).recurrenceInterval ?? 7)
  : "7",
```

The page serializer (`app/(app)/tasks/page.tsx`) does not serialize `recurrenceInterval` (confirmed at lines 56–68), so the value would always be `undefined`, the cast would yield `undefined ?? 7 → 7`, and the form would always default to 7 days for recurring tasks even if the original interval was different. The `Task` interface should include `recurrenceInterval?: number | null` and the page serializer should include the field.

#### Finding 8 — NON-BLOCKING: `postponeCount` and `estimatedMinutes` not serialized in tasks page

**`app/(app)/tasks/page.tsx` lines 56–68**

The `serializedTasks` mapping does not include `postponeCount` or `estimatedMinutes`. Both fields are declared as optional in the `Task` interface in `task-list.tsx`, so TypeScript does not catch the omission. The client will always receive `undefined` for these fields, meaning:

- The postpone-count badge on TaskItem will never render for tasks in the Tasks page (only renders when `postponeCount >= 1`)
- The estimated-minutes badge will never render

The topic detail page (`app/(app)/topics/[id]/page.tsx`, lines 49–63) correctly serializes both fields. The tasks page is missing them.

#### Finding 6 — NON-BLOCKING (safe as written): `coinsEarned` guard for zero-value tasks

**Lines 302–308**

```typescript
const refunded = data.task?.coinValue ?? 0;
if (refunded > 0) {
  window.dispatchEvent(
    new CustomEvent("coinsEarned", { detail: { delta: -refunded } })
  );
}
```

The `> 0` guard correctly prevents a `CustomEvent` with `delta: 0` or `delta: -0` from being dispatched for zero-value tasks. This is safe. The `CoinCounter` component presumably adds `delta` to the displayed balance — dispatching `delta: 0` would be harmless but noisy. No bug here, but the intent of the guard should be documented with a brief comment.

### `components/dashboard/daily-quest-card.tsx`

#### Finding 9 — NON-BLOCKING: Double `res.json()` consumption on error branch

**Lines 135–141**

```typescript
if (!res.ok) {
  const data = (await res.json()) as { error?: string };   // body consumed here
  console.error("Failed to complete quest:", data.error);
  return;
}

const data = (await res.json()) as CompleteResponse;       // second read — unreachable after !ok
```

This is structurally correct because the `return` statement on the error branch prevents the second `res.json()` from ever executing after the error branch. However, the `data` variable is re-declared in the outer scope of the success path, which creates a naming collision that TypeScript resolves by treating them as separate block-scoped variables. The code is functionally correct but the pattern could confuse readers and future contributors into thinking the second `res.json()` could be reached. A combined approach (read `json()` once before branching) would be cleaner.

## Verdict

**CHANGES REQUESTED**

Two BLOCKING issues must be resolved before the next deploy:

1. **Finding 1** — `POST /api/tasks/:id/complete` must validate the timezone string (length cap + type check) the same way the postpone route does. The fix is a one-line Zod check or an inline length guard.

2. **Finding 2** — The daily quest selection algorithm uses a private UTC-only `getTodayString()` instead of the timezone-aware `getLocalDateString()`. This means the overdue/due comparisons in `pickBestTask`, `selectDailyQuest`, and `forceSelectDailyQuest` operate on the server's UTC date, not the user's local date — defeating the purpose of the timezone work done in this session. At minimum, replace all calls to `getTodayString()` with `getLocalDateString()` (no-arg fallback is already correct); full fix threads the timezone parameter through `selectDailyQuest`.

The remaining NON-BLOCKING and NIT items are straightforward and should be addressed in the next cleanup pass.
