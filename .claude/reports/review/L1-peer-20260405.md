## Findings

### BLOCKING

#### 1. `createTask` still uses server UTC for RECURRING `nextDueDate` fallback
**File:** `lib/tasks.ts:148`

```typescript
nextDueDate: input.type === "RECURRING"
  ? (input.dueDate ?? new Date().toISOString().split("T")[0])
  : null,
```

The fix in `completeTask` correctly moved to `getLocalDateString(timezone)` for the recurrence advance calculation. However, `createTask` still falls back to `new Date().toISOString().split("T")[0]` (server UTC) when no `dueDate` is provided. For a user in e.g. UTC-5 creating a recurring task at 22:00 local time (= 03:00 next UTC day), the initial `nextDueDate` will be set to tomorrow's UTC date, making the task invisible to the daily quest algorithm for that local day.

The fix is consistent with what was already applied to `completeTask`: call `getLocalDateString()` for the fallback. The `createTask` function does not currently accept a `timezone` parameter, so the fallback will use server time (UTC in containers), which is acceptable — but the inconsistency should at minimum be documented in a comment, and ideally the API route for task creation should accept and thread through a `timezone` parameter the same way `complete` does.

**Remediation:**
```typescript
// In lib/tasks.ts — createTask signature and nextDueDate fallback:
export async function createTask(
  userId: string,
  input: CreateTaskInput,
  timezone?: string | null   // add parameter
): Promise<Task> {
  // ...
  nextDueDate: input.type === "RECURRING"
    ? (input.dueDate ?? getLocalDateString(timezone))
    : null,
```

### NON-BLOCKING

#### 2. `completeTask` transaction split introduces a TOCTOU window for streak + achievements
**File:** `lib/tasks.ts:332–360`

The streak update and achievement check were intentionally moved outside the main transaction to prevent a streak failure from rolling back the coin award. This is a valid trade-off and the comment explains it clearly. However, moving `checkAndUnlockAchievements` outside the transaction as well is a side-effect that was not mentioned. Between the end of the coin/level transaction and the achievement check, a concurrent completion on the same user could also call `checkAndUnlockAchievements` with overlapping `totalCompleted` values, leading to the same achievement being unlocked twice.

The previous code passed `tx` to both `updateStreak` and `checkAndUnlockAchievements`, which gave serializability guarantees. The new code passes `undefined` to both, losing that guarantee.

This is non-blocking because `checkAndUnlockAchievements` itself checks whether an achievement is already in `user_achievements` before inserting (preventing duplicate rows), so the worst outcome is a double-toast on the client — not a data corruption.

**Recommendation:** Add a comment near the achievement call explicitly noting the TOCTOU risk and why it is acceptable, so the next reviewer does not mistake it for an oversight.

#### 3. `selectDailyQuest` — `todayStart` does not use the user's timezone
**File:** `lib/daily-quest.ts:234–235`

```typescript
const today = getLocalDateString();    // no timezone arg
const todayStart = new Date(`${today}T00:00:00Z`);
```

Two issues exist here:

**a.** `getLocalDateString()` is called without a `timezone` argument. `selectDailyQuest` does not accept a timezone parameter. For UTC+ users near midnight this can produce the wrong date for quest deduplication (the "already completed today" check at line 247 and the cleanup query at line 266). This is a pre-existing issue that the fix partially addressed (appending `Z` to `todayStart` is an improvement over the local-time constructor) but the root problem — no timezone threading — was not solved.

**b.** Even with the `Z` suffix, the semantics are subtly wrong for UTC+ users. `today` is derived from `getLocalDateString()` without timezone, which resolves to the server's UTC date. So for a user in UTC+2 at 00:30 local time, `todayStart` will be "yesterday 00:00Z" — the quest deduplication will not find their midnight-crossing completion and may offer a new quest when the user expects to see the celebration state.

The complete fix requires passing `timezone` as a parameter to `selectDailyQuest` and threading it through to `getLocalDateString` and `pickBestTask`. This is a larger refactor, but without it the UTC fix is incomplete for UTC+ edge cases.

**Impact:** Affects UTC+ users completing daily quests within the first hours past midnight. Low frequency but observable.

#### 4. `drizzle/0004_messy_zodiak.sql` — missing newline at end of file
**File:** `drizzle/0004_messy_zodiak.sql:8`

The migration file has no trailing newline (`\ No newline at end of file` per the diff). While this does not affect execution, it is inconsistent with the project's conventions and most POSIX-compliant tools expect a trailing newline. The sha256 hash used by the migration runner includes the file content, so fixing this will invalidate the hash for any environments that have already seeded it. This is only worth fixing before the first production deployment of this migration.

### NITs

#### N1. `lib/client/coin-events.ts` — `typeof delta !== "number"` guard is redundant with TypeScript
**File:** `lib/client/coin-events.ts:20`

```typescript
if (typeof delta !== "number" || !isFinite(delta) || delta === 0) return;
```

Since the function signature is typed `delta: number`, the `typeof` check is dead code at runtime in TypeScript output. The `!isFinite` and `delta === 0` guards are the valuable ones. This is a pure nit — the guard does not harm correctness.

#### N2. `lib/date-utils.ts` — redundant `Intl.DateTimeFormat` re-format in `getLocalTomorrowString` / `getLocalYesterdayString`
**Files:** `lib/date-utils.ts:58–68`, `lib/date-utils.ts:93–103`

When a valid `timezone` is provided, both functions:
1. Call `getLocalDateString(timezone)` which already runs `Intl.DateTimeFormat("en-CA", { timeZone: timezone }).format(now)` internally.
2. Then parse the returned string back to year/month/day numbers.
3. Then construct a UTC Date from those numbers.
4. Then re-run `Intl.DateTimeFormat("en-CA", { timeZone: timezone }).format(tomorrowUtc)`.

Step 4 is unnecessary. Since `tomorrowUtc` is computed at UTC midnight from the already-local date, formatting it back with the user's timezone will produce the correct YYYY-MM-DD string — but the intermediate step is doing redundant formatting work. The result is always correct; this is only a performance nit given `Intl.DateTimeFormat` format calls are cheap.

#### N3. `messages/*.json` — i18n key for `back_to_topics` embeds a raw `←` arrow character
**Files:** `messages/de.json`, `messages/en.json`, `messages/fr.json`

```json
"back_to_topics": "← Back to Topics"
```

The arrow character is part of the translation value rather than being rendered separately in the component. This is inconsistent with how navigation arrows are usually handled (CSS `::before` or a separate icon element) and means translators cannot independently control the direction indicator. This is a minor UX/i18n nit only.

## Positive Observations

The following changes are well-executed and represent clear improvements over the prior state:

- **`lib/client/coin-events.ts`** — Centralising the `coinsEarned` custom event into a typed module with a named constant eliminates magic string duplication across four components. The NaN/Infinity guard is a good defensive measure.

- **`lib/date-utils.ts` calendar arithmetic fix** — Using `Date.UTC(y, m - 1, d + 1)` from the parsed local date string correctly handles month and year rollovers (e.g. 2026-01-31 + 1 = 2026-02-01) that the previous `setDate()` approach also handled, but the new approach is more explicit about UTC anchoring.

- **`lib/tasks.ts` — `completeTask` RECURRING timezone fix** — Threading `getLocalDateString(timezone)` into `nextDueDate` calculation is the correct fix for UTC− users who were getting next-due dates shifted by one day.

- **`lib/validators/index.ts` — `TimezoneSchema` extraction** — Promoting the schema to a shared location with a JSDoc comment is the right pattern. Using `safeParse` in the API route instead of bare `typeof` is more robust.

- **`components/tasks/task-form.tsx` — mobile layout fix** — The `fixed inset-0 flex flex-col` + `flex-1` scrollable fields area + `flex-shrink-0` sticky footer is the canonical approach for full-screen modal forms on mobile. Adding `type="button"` to the cancel `<button>` prevents accidental form submission — a real bug fix.

- **`drizzle/0004_messy_zodiak.sql` — backfill strategy** — Using a correlated subquery to backfill `total_tasks_created` from existing `tasks` rows is correct and safe. The comment explaining it mirrors the schema comment, which is good for future maintainers.

- **`scripts/migrate.mjs` improvements** — Per-entry `try/catch` with `process.exit(1)` is the right failure mode for a migration runner. The `SET statement_timeout = 30000` guard is a good defensive addition. The regex fix supporting hyphenated identifiers is a correctness improvement even if no current tables use hyphens.

- **`app/(app)/stats/page.tsx` icon fix** — Using `resolveTopicIcon` + `FontAwesomeIcon` with explicit `width`/`height` and `color: topic.color ?? "var(--accent-amber)"` is consistent with the icon rendering pattern used in `topic-card.tsx` and `topic-detail-view.tsx`.

- **`app/(app)/topics/[id]/page.tsx` i18n** — `Promise.all([getTopicById, getTranslations])` is correct for parallel async fetches and reduces the server render time vs. sequential awaits.
