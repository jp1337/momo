#### B-01: Docker Hub image name uses wrong username in merge job
**File:** `.github/workflows/build-and-publish.yml` — line 127
**Commit introduced:** bb4de8a

The `docker.io` image reference in the merge job's `metadata-action` uses `kermit1337/momo` while every other reference in the file (and the repository itself) uses `jp1337`. This causes the multi-arch manifest to be pushed to a different Docker Hub repository than intended (or to fail if `kermit1337` is not an accessible namespace under the `DOCKERHUB_USERNAME` secret).

```yaml
# Current (line 127):
docker.io/kermit1337/momo

# All other references in same file:
ghcr.io/jp1337/momo
quay.io/jp1337/momo
```

**Remediation:** Change `kermit1337/momo` to `jp1337/momo` on line 127 of the workflow.

#### B-02: Docker Hub and Quay.io logins are unconditional in merge job — will fail if secrets are absent
**File:** `.github/workflows/build-and-publish.yml` — lines 143–154
**Commit introduced:** bb4de8a (regressed the conditional guards from 35fa9e5)

Commit 35fa9e5 explicitly added `if: ${{ secrets.DOCKERHUB_USERNAME != '' }}` and `if: ${{ secrets.QUAY_USERNAME != '' }}` guards on both registry login steps, along with `enable=` conditions on the metadata-action images. The subsequent complete rewrite in bb4de8a removed all of those guards. The login steps now run unconditionally: if `DOCKERHUB_USERNAME` or `QUAY_USERNAME` are not set in the repository secrets, the docker/login-action will receive an empty username and fail the job. The `imagetools create` step will also attempt to push to all three registries regardless of which logins succeeded.

The prior commit message for 35fa9e5 explicitly states: "Docker Hub und Quay.io werden nur genutzt wenn DOCKERHUB_USERNAME bzw. QUAY_USERNAME als Secret gesetzt sind." This intent was lost in the rewrite.

**Remediation:** Re-add the `if:` conditions on the Docker Hub and Quay.io login steps, and add `enable=${{ secrets.DOCKERHUB_USERNAME != '' }}` / `enable=${{ secrets.QUAY_USERNAME != '' }}` to the metadata-action image entries as was done in 35fa9e5.

#### B-03: `DELETE /api/wishlist/:id/buy` returns 404 "Item not found" for a semantically distinct conflict error
**File:** `app/api/wishlist/[id]/buy/route.ts` — lines 64–69
**Commit introduced:** fc0bcf5

The `unmarkAsBought` function in `lib/wishlist.ts` throws `"Wishlist item is not marked as bought"` when the item exists but is in a non-BOUGHT state. The route's error classifier detects `error.message.includes("not marked")` and maps it to HTTP 404 with the message `"Item not found"`. The item does exist — returning 404 misleads the caller into thinking the item does not exist when the real problem is a state conflict (the item has already been un-bought or was never bought). HTTP 409 Conflict or 422 Unprocessable Entity with an accurate message would be correct. This can cause client-side logic that retries on 404 to loop or to incorrectly display "item not found" UI.

```typescript
// Current — maps "not marked" to 404 "Item not found":
const isNotFound =
  error instanceof Error &&
  (error.message.includes("not found") || error.message.includes("not marked"));
return Response.json(
  { error: isNotFound ? "Item not found" : "Internal server error" },
  { status: isNotFound ? 404 : 500 }
);

// Correct:
const isNotFound = error instanceof Error && error.message.includes("not found");
const isConflict = error instanceof Error && error.message.includes("not marked");
if (isConflict) {
  return Response.json({ error: "Item is not in a bought state" }, { status: 409 });
}
return Response.json(
  { error: isNotFound ? "Item not found" : "Internal server error" },
  { status: isNotFound ? 404 : 500 }
);
```

#### N-01: `actions/checkout@v6` and `actions/setup-node@v6` do not exist — will fail at runtime
**File:** `.github/workflows/docs.yml` — lines 22, 36 (also referenced in commit d6c002f message)
**Commit introduced:** d6c002f

The commit message states that actions were updated to v6 to address Node.js 20 deprecation warnings. As of the knowledge cutoff (August 2025), `actions/checkout` and `actions/setup-node` have not released a v6. The latest stable versions are v4. Using a non-existent tag will cause the workflow job to fail with "Unable to resolve action". The `build-and-publish.yml` correctly retained v4 for these actions; only `docs.yml` received the incorrect version bump.

**Remediation:** Revert `actions/checkout@v6` to `actions/checkout@v4` and `actions/configure-pages@v6` to `actions/configure-pages@v4` in `docs.yml`. Verify that `actions/deploy-pages@v5` and `actions/upload-pages-artifact@v4` are also valid before deploying.

#### N-02: `serialize-javascript` override uses an open-ended `>=` range
**File:** `package.json` — line 28
**Commit introduced:** 8fea82d

The override is specified as `"serialize-javascript": ">=7.0.5"`. The `>=` range allows npm to install any future major version (8.x, 9.x, ...) that may contain breaking API changes. The standard practice for security-fix overrides is to use `^7.0.5` (compatible within the same major) or an exact version. The `@rollup/plugin-terser` package (which pulls in serialize-javascript) declares a peer dependency of `^6.0.1`, so the mismatch is intentional but should be constrained to the same major to avoid unintended upgrades. Currently `package-lock.json` pins 7.0.5 and this is safe until `npm update` is run without a lock file.

**Remediation:** Change `">=7.0.5"` to `"^7.0.5"` to constrain the override to the 7.x series.

#### N-03: Static Keycloak import loads the OIDC provider module even when OIDC is not configured
**File:** `lib/auth.ts` — line 18
**Commit introduced:** 0254124

The previous implementation used a dynamic `require()` inside `buildProviders()` to load the Keycloak provider only when `OIDC_ISSUER` was set. The change replaces it with a top-level static `import`. While this is idiomatic and avoids the `require()` / TypeScript `any` concern, it does mean the Keycloak OIDC provider module is now imported unconditionally at server startup, even on deployments that only use GitHub/Discord/Google. The performance impact is negligible (it is a small module with no side effects at load time), but the original developer comment noted the intent: "Dynamic import to avoid loading provider code when OIDC is not configured." The behaviour change should be acknowledged in the commit message or a code comment, as the comment was simply deleted rather than updated.

**Remediation (non-blocking):** Add a brief comment above the Keycloak import noting that the module is always loaded and has no startup side effects, so the static import is safe:
```typescript
// Imported unconditionally; has no startup side effects when the provider is not enabled.
import Keycloak from "next-auth/providers/keycloak";
```

#### NIT-01: `eslint-disable-next-line` in `achievement-toast.tsx` suppresses only `setQueue` but not the immediately following `setCurrent` call
**File:** `components/animations/achievement-toast.tsx` — lines 114–116
**Commit introduced:** 0254124

The suppression comment on line 114 covers only line 115 (`setQueue(achievements)`). Line 116 (`setCurrent(achievements[0] ?? null)`) is a second synchronous `setState` call inside the same effect body and is equally subject to the `react-hooks/set-state-in-effect` rule. The ESLint run currently passes (verified by `npm run lint`), which suggests the rule's implementation treats consecutive setState calls in the same effect as a single diagnostic unit. However, relying on this implementation detail is fragile. Adding a second `// eslint-disable-next-line` before line 116 — or using a single `// eslint-disable react-hooks/set-state-in-effect` block comment around both lines — would make the suppression explicit and reviewer-intent clear.

#### NIT-02: `react-hooks/set-state-in-effect` suppression in `theme-toggle.tsx` uses an undocumented rule name convention
**File:** `components/theme-toggle.tsx` — line 33
**Commit introduced:** 0254124

The `react-hooks/set-state-in-effect` rule is part of the React Compiler hooks plugin (enabled via `eslint-config-next`), not the standard `eslint-plugin-react-hooks` shipped with React 18. Its inclusion in the config is not documented anywhere in the codebase. The inline suppression comment is correct and passes lint, but a brief note in `eslint.config.mjs` (or a code comment) explaining that `set-state-in-effect` comes from the React Compiler plugin would help future maintainers who encounter an unfamiliar rule name.
