## 1. Architectural Assessment by Area

### 1.1 `lib/client/` Subdirectory ŌĆö Appropriate Placement

The introduction of `lib/client/coin-events.ts` establishes a `lib/client/` namespace for modules that are imported by client components but must not import any server-side code (`lib/db`, `lib/env`, etc.). This is an appropriate and standard Next.js pattern. The module correctly contains no server imports ŌĆö only a `CustomEvent` dispatch that is safe to tree-shake on the server.

**Assessment: correct.** The `lib/CLAUDE.md` documents the module and its four call sites. Dependency direction is correct in all four places: `components/ ŌåÆ lib/client/`, never the reverse.

One minor note: the `typeof delta !== "number"` check on line 20 is dead code given the TypeScript signature, but it is a defensive pattern that causes no harm and the L1 review already flagged this as a nit.

### 1.2 `lib/db/schema.ts` ŌĆö `totalTasksCreated` Counter Column

The design decision to use an append-only counter column on the `users` table rather than a `COUNT(*)` from `tasks` at query time is architecturally correct for this use case. The stated invariant ŌĆö "incremented on `createTask()`, never decremented" ŌĆö is a well-understood immutable-counter pattern.

**Verification of invariant coverage:**

| Task creation path | Counter incremented? |
|---|---|
| `createTask()` ŌĆö single task via POST /api/tasks | Yes (line 158) |
| `breakdownTask()` ŌĆö N subtasks via POST /api/tasks/:id/breakdown | No (see section 2.1 ŌĆö BLOCKING) |
| Direct DB imports / admin seed | Not applicable (seed route uses `createTask()` indirectly) |

The counter increment is correctly placed inside the `createTask` transaction, so either both the task insert and the counter increment succeed or neither does. This is the correct atomicity boundary.

The backfill in `drizzle/0004_messy_zodiak.sql` uses a correlated `COUNT(*)` subquery which correctly seeds existing users from their current task count. The comment in `lib/statistics.ts:339` ("Existing users were backfilled to their task count at migration time") is accurate and useful.

**Assessment: mostly correct. One gap exists in `breakdownTask` (see Finding 2.1).**

### 1.3 `lib/tasks.ts` ŌĆö Transaction Boundary Restructuring

The prior architecture ran all of `completeTask` inside a single Drizzle transaction: coin award + streak update + achievement check. The current architecture splits this into three phases:

1. **Transaction:** task update + completion record + coin award + level detection
2. **Outside transaction (try/catch):** streak update
3. **Outside transaction:** completion count query + achievement check

The rationale documented in the code ŌĆö "a transient streak failure should not roll back the task completion or coin award" ŌĆö is a valid product decision. The failure modes are:

| Failure | Consequence (new) | Consequence (old) |
|---|---|---|
| Streak update fails | Task marked done, coins awarded; streak unchanged (non-fatal) | Full rollback ŌĆö task not marked done |
| Achievement check fails | Task done, coins awarded, streak updated | Full rollback |
| Coin award fails | Transaction rolls back; nothing persisted | Same |

The new behaviour is more user-friendly: the core action (task completion + coins) cannot be rolled back by a streak or achievement subsystem failure. This trade-off is documented and defensible.

**Structural concern:** moving `checkAndUnlockAchievements` outside the transaction was apparently a consequence of `updateStreak` being moved, not a deliberate decision. The two operations have different risk profiles:

- Streak failure is plausible (timezone edge cases, user row not found).
- Achievement failure under concurrency creates a TOCTOU window where two concurrent completions both read `totalCompleted = N` before either inserts into `user_achievements`, producing a double unlock toast (no data corruption ŌĆö the unique constraint prevents duplicate rows).

The L1 review flagged this. The L2 assessment adds: since `checkAndUnlockAchievements` already reads and writes `user_achievements` outside any transaction, the TOCTOU risk existed before this change whenever the function was called with `tx = undefined`. The change does not worsen the data integrity story (the insert is guarded by a pre-check + implicit deduplication), but a code comment documenting the intentional TOCTOU acceptance should be added at the call site.

**Assessment: acceptable trade-off, but the achievement check should have an explicit comment at line 350 documenting the TOCTOU acceptance.**

### 1.4 Timezone Threading ŌĆö Completeness Audit

The timezone-threading pattern requires that any function that compares against "today" receives an IANA timezone string from the client and passes it down to `getLocalDateString()`. The following audit covers all functions that use a date comparison:

| Function | Timezone parameter | Status |
|---|---|---|
| `completeTask` (tasks.ts) | `timezone?: string | null` | Correct ŌĆö threaded from POST /api/tasks/:id/complete |
| `updateStreak` (gamification.ts) | `timezone?: string | null` | Correct |
| `postponeDailyQuest` (daily-quest.ts) | `timezone?: string | null` | Correct ŌĆö threaded from POST /api/daily-quest/postpone |
| `getLocalTomorrowString` (date-utils.ts) | `timezone?: string | null` | Correct |
| `getLocalYesterdayString` (date-utils.ts) | `timezone?: string | null` | Correct |
| `selectDailyQuest` (daily-quest.ts) | **None** | Gap ŌĆö no timezone parameter |
| `forceSelectDailyQuest` (daily-quest.ts) | **None** | Gap ŌĆö no timezone parameter |
| `pickBestTask` (daily-quest.ts) | **None** | Falls back to server UTC |
| `createTask` (tasks.ts) ŌĆö RECURRING fallback | **None** | Gap ŌĆö no timezone parameter |

The `selectDailyQuest` and `forceSelectDailyQuest` gaps were flagged in the L1 review. The L2 assessment adds structural context: these are the only two exported functions in the `daily-quest` module that interact with date-sensitive logic but do not accept a `timezone` parameter. The route at `app/api/daily-quest/route.ts` would need to accept timezone from the request body and thread it through, mirroring the pattern established in the complete and postpone routes.

The `createTask` gap is the BLOCKING finding (section 2.1 below).

### 1.5 `lib/validators/index.ts` ŌĆö `TimezoneSchema` Extraction

**Assessment: incomplete ŌĆö one call site not updated.**

### 1.6 `lib/date-utils.ts` ŌĆö Calendar Arithmetic Fix

The fix for `getLocalTomorrowString` and `getLocalYesterdayString` is architecturally correct. The key insight is:

1. Parse today's string in the user's local calendar (via `getLocalDateString`).
2. Decompose to `[y, m, d]`.
3. Construct a UTC Date from those components with `d ┬▒ 1`.
4. Re-format with the user's timezone to get the correct local date string.

This correctly handles all DST transitions and month/year boundaries because the arithmetic is performed on calendar integers, not on a JavaScript `Date` object (which uses millisecond arithmetic and is therefore sensitive to DST shifts).

The L1 nit about redundant `Intl.DateTimeFormat` re-formatting is noted: when `timezone` is provided, step 4 re-runs `Intl.DateTimeFormat` on a UTC Date that was constructed from the already-local-date integers. For any valid timezone, formatting a UTC midnight date with the timezone will produce the same YYYY-MM-DD string as the arithmetic result. The re-format is safe but redundant; the fallback branch (no timezone) now correctly uses `getUTC*()` methods rather than local getters, which is an improvement over the previous code.

### 1.7 `selectDailyQuest` ŌĆö `todayStart` UTC Fix

```typescript
const today = getLocalDateString();            // no timezone ŌĆö server UTC
const todayStart = new Date(`${today}T00:00:00Z`);
```

The change from `T00:00:00` (local time constructor ŌĆö environment-dependent) to `T00:00:00Z` (UTC) is an improvement: it removes the environment-dependent behaviour where the server's local timezone would affect the parsed Date. However, `today` still comes from `getLocalDateString()` with no argument, which on production containers (UTC) returns the UTC date. The resulting `todayStart` is UTC midnight of the UTC date, which is correct for UTC users but wrong for UTC+ users past midnight.

For example: a user in UTC+2 completing their quest at 00:30 local time (= 22:30 previous UTC day) ŌĆö `today` will be the previous UTC date, `todayStart` will be the midnight of that date, and the "already completed today" check at line 247 will not find the completion from 22:30 UTC the prior day. The user will see an incorrect state.

This is a non-blocking issue because it requires a timezone parameter to fix properly, which is a larger refactor than this delta addressed. The `Z` suffix fix is a genuine improvement over the prior state.

## 2. Findings

### 2.1 BLOCKING ŌĆö `createTask` RECURRING `nextDueDate` Still Uses Server UTC

**File:** `lib/tasks.ts:148`
**Severity:** Blocking (correctness regression for UTCŌłÆ users, incomplete timezone fix)

The L1 review correctly identified this. The L2 perspective adds:

The timezone fix was applied to `completeTask` at the point where `nextDueDate` is *advanced* (recurrence interval calculation). The same fix was not applied to `createTask` where `nextDueDate` is *initialised*. For a RECURRING task with no explicit `dueDate`, the fallback:

```typescript
nextDueDate: input.type === "RECURRING"
  ? (input.dueDate ?? new Date().toISOString().split("T")[0])
  : null,
```

uses the server's UTC clock. In production containers running UTC this is only wrong for users in UTCŌłÆ at times where local date is behind UTC date (e.g. 23:00 UTC-5 = next day UTC). In those cases the initial `nextDueDate` is set to tomorrow's UTC date instead of today's local date, making the task invisible to the daily quest algorithm until the UTC date catches up.

**Required fix:** Add an optional `timezone` parameter to `createTask` and pass it from the POST /api/tasks route body (mirroring the complete route pattern):

```typescript
// lib/tasks.ts
export async function createTask(
  userId: string,
  input: CreateTaskInput,
  timezone?: string | null
): Promise<Task> {
  // ...
  nextDueDate: input.type === "RECURRING"
    ? (input.dueDate ?? getLocalDateString(timezone))
    : null,
```

```typescript
// app/api/tasks/route.ts ŌĆö POST handler
const body = await request.json() as { timezone?: unknown; [key: string]: unknown };
const tz = TimezoneSchema.safeParse((body as Record<string, unknown>)?.timezone);
const timezone = tz.success ? tz.data : null;
// ...
const task = await createTask(user.userId, parsed.data, timezone);
```

The `CreateTaskInputSchema` should either include the `timezone` field or the route should parse it separately (the latter is the existing pattern in the complete route and is preferable to polluting the task input schema with a routing concern).

### 2.2 NON-BLOCKING ŌĆö `breakdownTask` Does Not Increment `totalTasksCreated`

**File:** `lib/tasks.ts:527ŌĆō583`
**Severity:** Non-blocking (data drift in stats)

`breakdownTask` inserts N new tasks (subtasks) into the `tasks` table but does not update `users.totalTasksCreated`. The `createTask` path is the only path that increments the counter. When a user breaks down a task into 5 subtasks, the counter is not incremented, creating a discrepancy between the counter and the actual number of tasks ever created.

The function was not changed in this delta ŌĆö the oversight predates this commit. However, because this delta is where `totalTasksCreated` was introduced and defined as an invariant counter, this is the correct time to enforce the invariant across all creation paths.

**Impact:** `stats` page will undercount `totalTasksCreated` for users who use the breakdown feature.

**Required fix:** Add counter increment inside the `breakdownTask` transaction, using `sql\`${users.totalTasksCreated} + ${subtaskTitles.length}\`` to account for N tasks created in one call.

### 2.3 NON-BLOCKING ŌĆö `selectDailyQuest` Timezone Gap Remains Unthreaded

**File:** `lib/daily-quest.ts:231ŌĆō299`
**Severity:** Non-blocking (pre-existing, UTC+ edge case)

`selectDailyQuest` and `pickBestTask` use `getLocalDateString()` without a timezone argument. The "already completed today" deduplication check (`gte(tasks.completedAt, todayStart)`) uses a `todayStart` derived from the UTC date, which is incorrect for UTC+ users near midnight.

This was partially addressed by adding the `Z` suffix to the `todayStart` constructor. Full resolution requires threading a `timezone` parameter through `selectDailyQuest ŌåÆ pickBestTask`.

The route at `app/api/daily-quest/route.ts` does not accept a timezone from the request body. A consistent fix would:
1. Add `timezone?: string | null` to `selectDailyQuest` and `forceSelectDailyQuest` signatures.
2. Accept timezone in the GET and POST handlers of `app/api/daily-quest/route.ts`.
3. Pass it to all `getLocalDateString()` calls within `selectDailyQuest` and `pickBestTask`.

**Impact:** UTC+ users near midnight may see a new quest when they expect the completion celebration state, or vice versa.

### 2.4 NON-BLOCKING ŌĆö `PostponeBodySchema` Timezone Inline Not Replaced With `TimezoneSchema`

**File:** `app/api/daily-quest/postpone/route.ts:19`
**Severity:** Non-blocking (DRY inconsistency)

The `TimezoneSchema` export was created in `lib/validators/index.ts` to centralise the timezone validation constraint. The postpone route still uses its own inline definition:

```typescript
timezone: z.string().max(64).optional().nullable(),
```

If the max length constraint on `TimezoneSchema` is ever changed (e.g. to enforce IANA name length more accurately), the postpone route will silently diverge. This is the same type of magic string duplication the `coin-events.ts` refactor was intended to prevent.

**Required fix:** Replace the inline definition with `TimezoneSchema.unwrap()` or by spreading it into the `PostponeBodySchema` object:

```typescript
import { TimezoneSchema } from "@/lib/validators";

const PostponeBodySchema = z.object({
  taskId: z.string().uuid("taskId must be a valid UUID"),
  timezone: TimezoneSchema,
});
```

## 3. Pattern Consistency Assessment

### Timezone Threading Pattern

The established pattern is: API route reads `timezone` from request body, validates with `TimezoneSchema`, passes to `lib/` function which passes to `getLocalDateString()`. The pattern is correctly implemented in:

- POST /api/tasks/:id/complete ŌåÆ `completeTask` ŌåÆ `getLocalDateString(timezone)`
- POST /api/daily-quest/postpone ŌåÆ `postponeDailyQuest` ŌåÆ `getLocalDateString(timezone)`

The pattern is absent from:

- POST /api/tasks ŌåÆ `createTask` (BLOCKING gap)
- GET /api/daily-quest ŌåÆ `selectDailyQuest` (non-blocking gap)
- POST /api/daily-quest ŌåÆ `forceSelectDailyQuest` (non-blocking gap)

### Error Handling Pattern

All API routes follow the project convention: `try/catch` with `console.error` server-side, typed `{ error: string }` responses, no stack trace exposure. The `completeTask` function introduces a nested try/catch for the non-fatal streak update ŌĆö this is consistent with the "thin API route" pattern because the error handling is in the lib function, not in the route.

### Immutable Counter Pattern

The `totalTasksCreated` counter follows the correct immutable-counter pattern: atomic SQL increment (`sql\`${users.totalTasksCreated} + 1\``), never decremented, placed inside the same transaction as the row insert. The pattern is correctly documented in both the schema comment and the statistics fallback. The gap in `breakdownTask` breaks the invariant.

### Dependency Direction

All changed files maintain the correct dependency direction:
- `lib/client/coin-events.ts` imports nothing from `lib/` server modules
- `components/` import from `lib/client/`, not from `lib/` server modules directly
- `app/api/` imports from `lib/` business logic functions
- `lib/` business logic does not import from `app/` or `components/`

No violations found in this delta.

## 4. ADR Notes ŌĆö Streak-Outside-Transaction Decision

The decision to move `updateStreak` outside the `completeTask` transaction deserves an ADR or inline rationale. The trade-off:

**WHY:** A streak update failure (e.g. user row not found due to concurrent delete, timezone lookup failure, transient DB error) should not roll back a task completion and coin award. The user's primary action should always succeed.

**WHAT IS LOST:** Streak and achievement data may be slightly stale relative to the task completion under concurrent load. A duplicate achievement toast is possible (not data corruption).

**ALTERNATIVES CONSIDERED:**
- Keep all in one transaction: rejected ŌĆö streak failure undoes user's completion, bad UX.
- Separate streak into a background job: overkill for this scale, adds infrastructure complexity.
- Keep streak in transaction, move only achievements outside: cleaner boundary, but streak and achievement context are both post-completion concerns so moving both is simpler.

The current approach (both streak and achievements outside transaction, streak with try/catch) is pragmatically correct for this scale and user load. The decision should be documented in a comment block at the call site.

## 6. Positive Architectural Observations

- **`lib/client/coin-events.ts`:** Correct separation of client-safe modules into a dedicated namespace. Four components consistently import from one location. The typed contract eliminates magic string duplication.

- **`totalTasksCreated` counter design:** The append-only counter correctly addresses the "stat persists after deletion" requirement that a live `COUNT(*)` cannot provide. Placing the increment inside `createTask`'s transaction is the right atomicity boundary.

- **`completeTask` transaction split with documented rationale:** The decision is product-defensible and the comment explains the intent. The failure isolation is correct.

- **`date-utils.ts` calendar arithmetic:** `Date.UTC(y, m - 1, d ┬▒ 1)` from parsed local integers is the correct approach for DST-safe day arithmetic. The fallback branch now correctly uses `getUTC*()` methods.

- **`TimezoneSchema` extraction:** The named export with JSDoc and the max(64) constraint is the correct pattern. The `safeParse` usage in the complete route correctly replaces the prior `typeof` check.

- **Migration backfill strategy:** The correlated `COUNT(*)` subquery correctly handles zero-task users (produces 0) and multi-task users. The column default of 0 is correct for new users created after the migration.
