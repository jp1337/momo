## Executive Summary

This review is scoped to the CI/CD pipeline rewrite introduced in commit bb4de8a and the supporting dependency management changes. The primary architectural change is the replacement of a single QEMU-emulated multi-arch build job with a three-job native matrix pipeline (lint → build[matrix] → merge). The pattern is sound and represents a significant improvement in build correctness and speed.

Two architectural deficiencies in the new pipeline were identified. One pre-existing finding from L2-arch-20260331 (the absence of a lint/typecheck gate) is now closed by the new `lint` job. All other prior findings from L2-arch-20260331 remain open and are not re-examined here.

## 1. Pipeline Architecture Assessment

### 1.1 Digest-Based Multi-Arch Build Pattern

The rewritten workflow implements the canonical Docker multi-arch build pattern using `push-by-digest=true`:

1. Each platform is built natively on a matching runner (amd64 on `ubuntu-latest`, arm64 on `ubuntu-24.04-arm`).
2. Each build job pushes the platform-specific image to GHCR without a tag, receiving only a content-addressable digest reference (`sha256:...`).
3. The digest is written as a filename to `/tmp/digests/` and uploaded as a GitHub Actions artifact with `retention-days: 1`.
4. The `merge` job downloads both artifacts, reconstructs the two digest references, and calls `docker buildx imagetools create` to assemble a multi-arch manifest list that is pushed to all target registries simultaneously.

This is architecturally correct. The key properties that make it correct:

- **No QEMU emulation.** ARM64 layers are built natively, eliminating the 3–10x slowdown from emulation. This makes the pipeline viable for development-pace pushes to `main`.
- **Digest immutability.** The digest references passed to `imagetools create` point to immutable layer sets. There is no possibility of a race condition where a tag is overwritten between the build step and the merge step.
- **Platform cache isolation.** The GHA build cache is scoped per platform using `scope=${{ steps.platform.outputs.suffix }}` (resolves to `linux-amd64` and `linux-arm64`). Without this scoping, the two build jobs would compete to write cache entries with the same key, causing cache poisoning for one platform.
- **Artifact TTL.** Setting `retention-days: 1` on digest artifacts is appropriate. The artifacts are ephemeral inter-job data; there is no value in retaining them beyond the workflow run.

### 1.2 Lint Gate

The prior L2 review (section 9.4, recommendation 10) flagged the absence of a lint/typecheck job as a gap. The new workflow opens with a `lint` job that runs `npx tsc --noEmit` and `npm run lint` before the build matrix is started (`needs: lint`). This closes that finding. The implementation is correct:

- Uses `actions/setup-node@v4` with `cache: 'npm'` so `npm ci` is not re-run without cause.
- TypeScript type-checking (`tsc --noEmit`) runs before ESLint, which is the correct order — a type error that manifests as an implicit `any` would also fail lint, but the TypeScript error message is more precise.
- `node-version: '20'` matches the `FROM node:20-alpine` in the Dockerfile. This version consistency is important: a linter run on Node 20 will not produce different results than the image build.

**Closed finding from L2-arch-20260331:** Section 9.4 recommendation to add a CI lint/typecheck job as a required build gate is now satisfied.

### 1.3 fail-fast: false in Build Matrix

The `strategy.fail-fast: false` setting on the build job is deliberate and correct for this use case. If the arm64 build fails, the amd64 build is not cancelled. This allows:

- Faster diagnosis: both platform build logs are available when the merge job eventually fails.
- Potential partial recovery: if one platform's failure is transient (runner availability, network timeout), re-running only the failed matrix leg is possible.

The tradeoff is that a real build failure on one platform will waste the full build time of the other platform before the pipeline terminates. For a two-platform matrix where arm64 builds take longer, `fail-fast: true` would be faster to fail on amd64 errors. However, given that the most common failures in this codebase are TypeScript/lint errors (now caught by the upstream `lint` job), the remaining class of build-job failures is dominated by Dockerfile errors that would affect both platforms identically. The `fail-fast: false` choice is defensible.

### 1.4 Artifact Sharing Mechanism

The digest is communicated between jobs via file-based artifacts (upload-artifact / download-artifact). The implementation uses a naming convention where each artifact is named `digests-{platform-suffix}` and the `merge` job downloads all artifacts matching `digests-*` into a merged directory using `merge-multiple: true`.

This is correct. The alternative (passing digests via job outputs using `needs.build.outputs.*`) would require a separate step to aggregate matrix outputs, which is more complex in GitHub Actions because matrix job outputs are not natively aggregated — only the last matrix leg's output survives. The file-based approach avoids this limitation cleanly.

One implicit assumption: the digest file written by `touch "/tmp/digests/${digest#sha256:}"` relies on the Bash `${parameter#word}` parameter expansion to strip the `sha256:` prefix, then reconstructs it in the `imagetools create` step with `printf 'ghcr.io/jp1337/momo@sha256:%s ' *`. This round-trip is correct as long as:
- The digest always starts with `sha256:` (true for all current OCI runtimes).
- No other files are present in `/tmp/digests/` when `imagetools create` runs (guaranteed by `mkdir -p /tmp/digests` before the touch, combined with a fresh runner).

The `if-no-files-found: error` guard on the upload step ensures a failed build that produces no digest will fail immediately rather than silently passing an empty artifact to the merge job. This is a correct defensive measure.

## 2. Architectural Deficiencies in the New Pipeline

### 2.1 Missing Conditional Guards for Secondary Registry Logins (B-02 Elevation)

**Severity: Blocking — pipeline will fail if DOCKERHUB_USERNAME or QUAY_USERNAME are not set**

The L1 review (B-02) correctly identified that commit 35fa9e5 intentionally added `if:` conditions on the Docker Hub and Quay.io login steps and `enable=` expressions on the metadata-action image entries, and that commit bb4de8a removed them.

From an architecture standpoint, this represents a regression in the pipeline's operational contract. The design intent — stated explicitly in the 35fa9e5 commit message — is that GHCR is the always-on registry (backed by `GITHUB_TOKEN`, which is always available), while Docker Hub and Quay.io are opt-in registries activated by the presence of their respective secrets. This is the correct model for a project that may be forked or run in environments where only GHCR is configured.

The current unconditional model creates a hard dependency between pipeline success and the configuration of three external registry credentials. A developer forking this repository cannot run the CI pipeline without first creating Docker Hub and Quay.io accounts and configuring four secrets (`DOCKERHUB_USERNAME`, `DOCKERHUB_TOKEN`, `QUAY_USERNAME`, `QUAY_TOKEN`). This is an unnecessary coupling.

The correct architecture is:

```
GHCR login        → always executes (GITHUB_TOKEN is implicit)
Docker Hub login  → executes only if: secrets.DOCKERHUB_USERNAME != ''
Quay.io login     → executes only if: secrets.QUAY_USERNAME != ''

metadata-action images:
  name=ghcr.io/jp1337/momo
  name=docker.io/jp1337/momo,enable=${{ secrets.DOCKERHUB_USERNAME != '' }}
  name=quay.io/jp1337/momo,enable=${{ secrets.QUAY_USERNAME != '' }}
```

When a registry's `enable` expression evaluates to false, `docker/metadata-action` omits that registry from `DOCKER_METADATA_OUTPUT_JSON`. The `imagetools create` command therefore receives no `-t` flags for that registry and does not push to it. The login step not executing means no authentication failure occurs even when credentials are absent.

Note: the `imagetools create` command on line 161–163 constructs its `-t` flags exclusively from `DOCKER_METADATA_OUTPUT_JSON`. If the metadata-action correctly omits a registry due to `enable=false`, the push to that registry will not happen even if the login step somehow ran. However, the login steps will still fail unconditionally if the secrets are empty, because `docker/login-action` treats an empty username as an authentication error rather than a no-op. The `if:` guard on the login steps is therefore required independently of the metadata-action `enable=` expression.

### 2.2 Docker Hub Image Name Mismatch (B-01 Elevation)

**Severity: Blocking — pushes to wrong Docker Hub repository**

L1 B-01 identified that line 127 of the workflow references `docker.io/kermit1337/momo` while all other references use `jp1337`. This is an architectural correctness issue: the multi-arch manifest assembled by the `merge` job will be pushed to a Docker Hub namespace that is either inaccessible (if `kermit1337` is not the owner of the `DOCKERHUB_USERNAME` secret) or to a different repository than the one users are directed to pull from.

The inconsistency also creates a split-brain state: GHCR (`ghcr.io/jp1337/momo`) and Quay.io (`quay.io/jp1337/momo`) will receive correct multi-arch manifests, while the Docker Hub target will differ. If the `DOCKERHUB_USERNAME` secret is set to `jp1337`, the push to `kermit1337/momo` will fail with a 403. If the secret is set to `kermit1337`, the push will succeed to the wrong namespace, and `jp1337/momo` on Docker Hub will be stale.

This is a single-line correction (`kermit1337` → `jp1337`) but carries production impact.

## 3. docs.yml Action Version Problem (N-01 Elevation)

**Severity: Non-blocking on build-and-publish, blocking on docs deployment**

L1 N-01 identified that `docs.yml` references `actions/checkout@v6` and `actions/configure-pages@v6`, neither of which exists as of the knowledge cutoff. The architecture implication is that the documentation deployment pipeline is permanently broken on the current commit. Any push to `main` will trigger a failing workflow run.

This has no impact on the Docker build pipeline reviewed above, but it does mean the Jekyll-based docs site (introduced in Phase 8, `f2388b8`) has not successfully deployed since commit `d6c002f` updated the action versions. The `docs-site/` directory exists with a complete Jekyll configuration, and the workflow is structurally sound (Ruby setup, Jekyll build, Pages artifact upload, Pages deploy) — only the version tags are wrong.

The correct versions are `actions/checkout@v4` and `actions/configure-pages@v4`. The other actions in the file (`ruby/setup-ruby@v1`, `actions/upload-pages-artifact@v4`, `actions/deploy-pages@v5`) should be verified against the GitHub Actions marketplace before the workflow is corrected.

## 4. Dependency Management Strategy Assessment (N-02)

**Severity: Low risk, correctness improvement available**

The `package.json` `overrides` block uses two entries to enforce security-patched versions of transitive dependencies:

```json
"overrides": {
  "serialize-javascript": ">=7.0.5",
  "lodash": "4.17.21"
}
```

The strategies are inconsistent:

- `lodash` uses an exact pinned version (`4.17.21`). This is correct for a security override: it forces exactly the patched version and will not drift.
- `serialize-javascript` uses an open-ended range (`>=7.0.5`). This allows npm to resolve any version from 7.0.5 upward, including hypothetical 8.x or 9.x major releases that may introduce breaking API changes in `@rollup/plugin-terser`'s use of the library.

The `>=` range is currently safe because `package-lock.json` pins the resolved version to 7.0.5. The risk manifests when:
1. A developer runs `npm update` without `--save-exact`, or
2. A `npm ci` in a CI environment after `package-lock.json` is deleted or re-generated.

In either case, npm will resolve `>=7.0.5` to the latest available version at that time, which may be a major version that breaks the build.

The correct strategy is `^7.0.5` (semver compatible, same major), aligning with the lodash approach of specifying the minimum acceptable version within a known-compatible series. This is a dependency management correctness issue, not a functional bug in the current state.

## 5. Cache Architecture for Multi-Platform Builds

The GHA build cache scoping (`scope=linux-amd64` and `scope=linux-arm64`) is architecturally sound, but one property deserves explicit documentation: the GHA cache is keyed per repository, per branch, and per scope string. The implication is:

- A push to `main` that modifies only application code (not the Dockerfile or package.json) will get a cache hit on both platforms for the `deps` and `builder` stages, and will only re-execute the changed layers.
- A push that modifies `package.json` or `package-lock.json` will invalidate the `deps` stage cache for both platforms simultaneously, doubling the cache miss cost compared to the prior single-job QEMU approach. This is the correct trade-off: the cold-build time for native arm64 is still substantially faster than QEMU-emulated arm64.
- The `cache-to: type=gha,mode=max` setting writes all intermediate layers to cache, not just the final image. This maximises cache hit rate at the cost of increased cache storage usage. For a Next.js application with a multi-stage Dockerfile, the `deps` stage (npm install) is the most expensive and most likely to hit. `mode=max` is the correct setting.

No issues with the cache configuration. This section is informational.
