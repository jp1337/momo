## Summary

The codebase is well-structured and follows the project conventions documented in CLAUDE.md consistently. The thin-route / thick-lib separation is respected throughout, Zod validation is applied on all mutation endpoints, and every function has a JSDoc comment. The major findings are concentrated in three areas: race conditions in multi-step database operations, a non-atomic coin decrement during undo that can go negative in an edge case, and several logic gaps in the push notification fan-out code. There are no outright data-corruption bugs in the happy path, but the race conditions become real problems in concurrent usage.

## BLOCKING Findings

### B-01 ‚Äî Race condition in `completeTask`: non-atomic coin increment
**File:** `/home/jpylypiw/projects/momo/lib/tasks.ts` ‚Äî `incrementUserCoins` (lines 383‚Äì402) and `completeTask` (lines 215‚Äì308)

`completeTask` reads the task row, then separately reads the user's coin balance, adds to it, and writes it back in a read-modify-write sequence with no transaction or DB-level lock. If two requests complete tasks for the same user simultaneously (e.g., rapid double-tap or two browser tabs), both reads will see the same coin balance and one coin award will be silently lost.

The same pattern applies to `decrementUserCoins` in `uncompleteTask`.

**Remediation:** Use a single atomic SQL update instead of the read-modify-write:

```typescript
// Replace the read-modify-write with an atomic increment:
await db
  .update(users)
  .set({ coins: sql`${users.coins} + ${amount}` })
  .where(eq(users.id, userId));
```

This also eliminates the need for the `incrementUserCoins` helper to return the new value via a separate SELECT; use `.returning({ coins: users.coins })` on the UPDATE instead.

### B-02 ‚Äî Race condition in `completeTask`: non-atomic level-up detection
**File:** `/home/jpylypiw/projects/momo/lib/tasks.ts` lines 269‚Äì279

After the non-atomic coin increment, the code computes the level before/after in JavaScript by calling `getLevelForCoins(newCoins - coinsEarned)` and `getLevelForCoins(newCoins)`. Because `newCoins` was already derived from an unsafe read, and because the level written to DB (`users.level`) is updated separately from coins, there is a window where the DB `level` column can diverge from the level implied by `coins`. On a concurrent completion burst the level write may be silently skipped.

**Remediation:** Wrap the entire `completeTask` block (task update, completion insert, coin increment, level update) in a single Drizzle transaction. Use the `.returning()` value of the coin UPDATE to compute level-up without a second SELECT.

### B-03 ‚Äî Non-atomic streak + achievement write in `completeTask`
**File:** `/home/jpylypiw/projects/momo/lib/tasks.ts` lines 283‚Äì299, `lib/gamification.ts` lines 201‚Äì253

`updateStreak` and `checkAndUnlockAchievements` are called sequentially and each performs separate DB reads and writes. A partial failure (e.g., connection drop after streak write but before achievement write) leaves the data in an inconsistent state with no rollback path.

**Remediation:** Wrap `completeTask` in a transaction. All writes ‚Äî task update, completion insert, coin increment, level update, streak update, achievement inserts ‚Äî should either all succeed or all roll back.

### B-04 ‚Äî `sendStreakReminders` double-queries all completions per user, then discards the result
**File:** `/home/jpylypiw/projects/momo/lib/push.ts` lines 259‚Äì334

The implementation first runs a broad query to check if any completion rows exist for the user (line 262‚Äì275), then calls `checkCompletedToday` which performs two more identical queries and filters in JavaScript (lines 309‚Äì334). The `completionsToday` variable at line 276 checks only `isNotNull(completedAt)` ‚Äî every user with any completions will pass this check, defeating the intended early-exit optimisation. The actual today-filtering is then duplicated in `checkCompletedToday`.

More critically, `checkCompletedToday` fetches all completions for the user with no date filter at all (line 323‚Äì329, `isNotNull(completedAt)` only), then filters in JS. For a user with hundreds of completions this loads the entire history into memory on every cron tick.

**Remediation:** Replace the two-stage approach with a single query that includes a `gte(taskCompletions.completedAt, todayStart)` filter directly in Drizzle. The `todayStart` parameter passed to `checkCompletedToday` is never used in the query.

### B-05 ‚Äî `POST /api/admin/seed` is not admin-only
**File:** `/home/jpylypiw/projects/momo/app/api/admin/seed/route.ts` lines 19‚Äì35

The route only checks `!session?.user?.id` ‚Äî any authenticated user can call it. While the operation is idempotent and `seedAchievements` uses `onConflictDoNothing`, this is still an unprotected admin action. An attacker with a valid session can use it to verify achievement key names, and future additions to the seed function may not be idempotent.

**Remediation:** Add an admin check. At minimum, gate on `NODE_ENV === "development"`. A proper solution is an `isAdmin` flag on the users table or an allowlist of admin email addresses checked here.

## HIGH Findings

### H-01 ‚Äî `forceSelectDailyQuest` duplicates the entire priority algorithm verbatim
**File:** `/home/jpylypiw/projects/momo/lib/daily-quest.ts` lines 231‚Äì327

`forceSelectDailyQuest` is an exact copy of the four-step priority query block from `selectDailyQuest` (lines 140‚Äì221). Any bug fixed or enhancement made to one will not be reflected in the other. The difference between the two is one early-return guard (`if (existing) return existing`).

**Remediation:** Extract the priority algorithm into a private `runPriorityAlgorithm(userId, today)` function and call it from both public functions.

### H-02 ‚Äî Cron routes are accessible without any secret when `CRON_SECRET` is unset
**File:** `/home/jpylypiw/projects/momo/app/api/cron/daily-quest/route.ts` lines 19‚Äì29, `/home/jpylypiw/projects/momo/app/api/cron/streak-reminder/route.ts` lines 19‚Äì29

```typescript
if (cronSecret) {          // <-- skips auth entirely when CRON_SECRET is not set
  ...
}
```

When `CRON_SECRET` is not configured (it is an optional env var), the endpoint has no authentication at all. Any HTTP client can trigger a fan-out of push notifications to all users. In production this is a denial-of-service vector and in a multi-tenant context could spam all users.

**Remediation:** Make `CRON_SECRET` required in `serverEnvSchema` (change `emptyToUndefined` to `z.string().min(16)`), or at minimum reject the request with 401 when no secret is configured rather than allowing unauthenticated access.

### H-03 ‚Äî `postponeDailyQuest` uses `toISOString().split("T")[0]` to generate the date
**File:** `/home/jpylypiw/projects/momo/lib/daily-quest.ts` line 362

```typescript
const tomorrowStr = tomorrow.toISOString().split("T")[0]; // YYYY-MM-DD
```

`toISOString()` always returns UTC. If the server is running in a non-UTC timezone (or a user in UTC-N submits this request just before midnight), "tomorrow" will be calculated as today in the user's local time. This is inconsistent with `getTodayString()` in the same file (line 39‚Äì44) which correctly uses local time getters (`getFullYear`, `getMonth`, `getDate`).

The same issue exists in `completeTask` in `lib/tasks.ts` at line 238:
```typescript
const nextDueStr = nextDue.toISOString().split("T")[0];
```

**Remediation:** Use the same local-time formatting pattern used by `getTodayString()` in both locations, or establish a consistent server timezone via `TZ=UTC` in the deployment environment and document this requirement.

### H-04 ‚Äî `getUserTopics` loads all tasks for the user to count them in JS
**File:** `/home/jpylypiw/projects/momo/lib/topics.ts` lines 51‚Äì69

```typescript
const taskRows = await db.select().from(tasks).where(eq(tasks.userId, userId));
```

This fetches every column of every task owned by the user to compute topic counts in JavaScript. For a user with hundreds of tasks this is significant over-fetching. The query runs on every topic list load (dashboard + topics page).

**Remediation:** Use a GROUP BY aggregate query:

```typescript
const counts = await db
  .select({
    topicId: tasks.topicId,
    total: count(),
    completed: sql<number>`COUNT(*) FILTER (WHERE ${tasks.completedAt} IS NOT NULL)`,
  })
  .from(tasks)
  .where(and(eq(tasks.userId, userId), isNotNull(tasks.topicId)))
  .groupBy(tasks.topicId);
```

### H-05 ‚Äî `updateStreak` uses `date` column string comparison that is timezone-sensitive
**File:** `/home/jpylypiw/projects/momo/lib/gamification.ts` lines 219‚Äì239

`streakLastDate` is a Postgres `date` column. Drizzle returns it as a `string` in `YYYY-MM-DD` format. The comparison `streakLastDate === today` works correctly as a string comparison only if both the DB value and `getTodayString()` use the same timezone. If the server's local timezone and the DB timezone differ, a streak completed just after midnight may be double-counted or missed.

**Remediation:** Enforce `TZ=UTC` in the process environment and change `getTodayString()` to use UTC methods (`getUTCFullYear`, etc.) to be unambiguous.

### H-06 ‚Äî `checkAndUnlockAchievements` has no duplicate-insert guard at DB level
**File:** `/home/jpylypiw/projects/momo/lib/gamification.ts` lines 290‚Äì344

The function fetches existing achievements, filters new ones, then inserts. Under concurrent task completion (two simultaneous completions triggering the same achievement), both reads may see 0 existing achievements and both inserts will succeed, creating duplicate `user_achievements` rows.

**Remediation:** Add a unique constraint on `(userId, achievementId)` in the schema and use `.onConflictDoNothing()` on the insert, or move the entire check inside the transaction from B-03.

### N-01 ‚Äî `lib/auth.ts` uses `require()` for the OIDC provider inside an ES module file
**File:** `/home/jpylypiw/projects/momo/lib/auth.ts` lines 62‚Äì68

```typescript

```

The file is a `.ts` module using ES imports everywhere else. The `require()` call was added to enable "dynamic import" but `require` is not dynamic in the ES module sense ‚Äî it is synchronous and will be bundled at build time anyway. More importantly, the `/* eslint-disable-line */` comment at line 25 suppresses a legitimate `@typescript-eslint/no-explicit-any` warning that hides a type-safety gap on the providers array.

**Remediation:** Use a conditional dynamic `import()` if lazy loading is truly desired, or just use a static import guarded by the environment check at the call site. Type the providers array as `Provider[]` from `next-auth/providers`.

### N-02 ‚Äî `handleTimeChange` in `NotificationSettings` sends malformed request
**File:** `/home/jpylypiw/projects/momo/components/settings/notification-settings.tsx` lines 208‚Äì229

When the user changes the notification time while already subscribed, `handleTimeChange` POSTs to `/api/push/subscribe` with only `{ notificationTime: newTime }` ‚Äî omitting the required `subscription` field. The server-side `subscribeBodySchema` requires `subscription.endpoint` and `subscription.keys`, so this POST will fail validation (422) every time. The failure is silently caught. The comment acknowledges this is broken ("time change requires re-subscribe") but the broken code is still shipped.

**Remediation:** Either remove the `handleTimeChange` API call entirely (as the comment suggests, it's non-functional) or implement a dedicated PATCH endpoint for updating notification time without needing the subscription object again.

### N-03 ‚Äî `UpdateTaskInputSchema` refinement misses the `undefined` case for RECURRING type change
**File:** `/home/jpylypiw/projects/momo/lib/validators/index.ts` lines 91‚Äì101

The `refine` on `UpdateTaskInputSchema` only blocks `recurrenceInterval === null` when `type === "RECURRING"`. It does not block the case where a task is changed to `RECURRING` but `recurrenceInterval` is `undefined` (not provided). A PATCH like `{ type: "RECURRING" }` with no `recurrenceInterval` will pass validation and create a `RECURRING` task with `null` recurrenceInterval. The `completeTask` function at line 235 then falls back to `task.recurrenceInterval ?? 1` which silently uses 1 day instead of rejecting.

**Remediation:**
```typescript
.refine((data) => {
  if (data.type === "RECURRING" &&
      (data.recurrenceInterval === null || data.recurrenceInterval === undefined)) {
    return false;
  }
  return true;
}, ...)
```

### N-04 ‚Äî `completeTask` level-up detection computes `levelBefore` incorrectly
**File:** `/home/jpylypiw/projects/momo/lib/tasks.ts` lines 270‚Äì272

```typescript
const levelBefore = await getUserLevel(userId, newCoins - coinsEarned);
const levelAfter = getLevelForCoins(newCoins);
const newLevel = levelAfter.level > levelBefore ? levelAfter : null;
```

`getUserLevel` is an `async` function that returns a `number` (level number), but `newLevel` is computed by comparing `levelAfter.level > levelBefore` where `levelBefore` is that number. This is numerically correct. However `getUserLevel` accepts `_userId` (unused) and `coins`, and computes the level purely from coins ‚Äî it makes no DB call. Marking it as `async` when it simply delegates to a pure synchronous function creates a misleading signature and an unnecessary microtask allocation on every completion.

**Remediation:** Make `getUserLevel` synchronous or inline the `getLevelForCoins` call directly.

### N-05 ‚Äî `getBudgetSummary` calculates monthly spend based on `createdAt`, not purchase date
**File:** `/home/jpylypiw/projects/momo/lib/wishlist.ts` lines 302‚Äì318

The query filters on `gte(wishlistItems.createdAt, startOfMonth)`. This means an item created in the previous month but marked bought this month will not count towards this month's spending. An item created this month but marked bought next month will count immediately upon creation. The column name and intention are misaligned ‚Äî it should filter on the date the item was bought (which is not stored) or use a dedicated `boughtAt` timestamp.

**Remediation (short term):** Add a `boughtAt timestamp` column to `wishlist_items` that is set when `status` transitions to `BOUGHT`, and filter on that column in `getBudgetSummary`. **Short term:** document the known limitation in a code comment.

### N-06 ‚Äî `assignDailyQuest` re-fetches the task after updating it without scoping by userId
**File:** `/home/jpylypiw/projects/momo/lib/daily-quest.ts` lines 402‚Äì410

```typescript
const updatedRows = await db
  .select()
  .from(tasks)
  .where(eq(tasks.id, task.id))   // no userId filter
  .limit(1);
```

The preceding UPDATE is correctly scoped with both `task.id` and `userId`, but the re-fetch does not include a `userId` filter. While in practice task IDs are UUIDs and cannot be guessed, this is a defense-in-depth gap ‚Äî if a task ID were somehow known to an attacker, data from another user could theoretically be returned here.

**Remediation:** Add `eq(tasks.userId, userId)` to the re-fetch WHERE clause.

### N-07 ‚Äî `rate-limit.ts` store is never pruned ‚Äî memory leak in long-running processes
**File:** `/home/jpylypiw/projects/momo/lib/rate-limit.ts` lines 21‚Äì59

The in-memory `Map` accumulates one entry per unique key (e.g., `"tasks-create:<userId>"`). Entries are logically expired when `now >= entry.resetAt` but are never deleted from the Map. In a process with many users the Map will grow unboundedly. The comment about multi-replica deployments is present, but the memory leak is not acknowledged.

**Remediation:** Either periodically prune stale entries (e.g., `setInterval` that deletes entries where `Date.now() >= resetAt`) or switch to a TTL-aware store. For the short term, add a size cap.

### N-09 ‚Äî `completeTask` does not clear `isDailyQuest` flag on task completion
**File:** `/home/jpylypiw/projects/momo/lib/tasks.ts` lines 246‚Äì256

When a ONE_TIME or DAILY_ELIGIBLE task is completed, `completedAt` is set but `isDailyQuest` is left as-is. The `selectDailyQuest` function clears `isDailyQuest` on completed tasks the *next time* it runs (line 120‚Äì129 of `daily-quest.ts`), but between a completion and the next `GET /api/daily-quest` call, `getDailyQuestIncludingCompleted` will still return the completed task as the "daily quest" ‚Äî which is technically correct. **Remediation:** Set `isDailyQuest: false` in the `completeTask` UPDATE for non-recurring tasks, or ensure the cleanup in `selectDailyQuest` is also called as part of the completion flow.

- The `Response.json` vs `NextResponse.json` inconsistency (push routes use `NextResponse`, others use `Response`) is cosmetic but worth normalising

The one structural inconsistency worth noting: `POST /api/daily-quest` (force-reselect) has no rate limit, while `POST /api/daily-quest/postpone` has one set to 10/minute. Apply rate limiting to `POST /api/daily-quest` for consistency (see N-08).
