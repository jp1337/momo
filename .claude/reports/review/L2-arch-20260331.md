**Date:** 2026-03-31
**Reviewer:** Claude Sonnet 4.6 (architecture agent)
**Mode:** system
**Scope:** Full project — all phases 1–8 (app/, lib/, components/, deploy/, .github/)
**L1 Input:** L1-peer-20260331.md (findings confirmed and elevated where architecturally relevant)

## Executive Summary

Momo is a well-organised Next.js 15 App Router application with a clearly articulated separation of concerns: thin API routes delegating to a thick `lib/` layer, Drizzle ORM enforcing schema ownership, and Zod providing boundary validation. The layering decision is sound and the code consistently follows it.

Three systemic architectural weaknesses exist that transcend individual bugs:

1. **Stateful gamification without transactions.** The `completeTask` flow mutates five separate DB tables (tasks, task_completions, users.coins, users.level, user_achievements) across multiple sequential round-trips with no wrapping transaction. This is not a code quality problem — it is an architectural mismatch between a multi-step domain operation and a schema that has no mechanism to enforce atomicity on it.

2. **Single-process state in a multi-replica deployment target.** The rate limiter, the in-memory `Map`, and the daily-quest flag (`is_daily_quest`) stored as a boolean column are all designed for single-process operation. The Kubernetes manifest deploys two replicas with anti-affinity, which means these assumptions will silently break under the intended production topology.

3. **Date arithmetic is untethered from timezone.** The app stores due dates and streak dates as bare `date` columns and computes "today" using server local time. No timezone constraint is enforced anywhere in the stack. The deployment manifests contain no `TZ=UTC` setting. This is an architecture gap, not just a code smell.

The sections below expand on each area, then cover secondary concerns.

## 1. App Router Alignment

### 1.1 Route Structure

The App Router grouping is correctly applied:

- `(app)/` for authenticated pages with a shared layout guard
- `(auth)/` for unauthenticated pages
- `api/` for API routes co-located with the app

The `params` handling in dynamic routes uses the Next.js 15 `Promise<{ id: string }>` signature correctly throughout (`app/api/tasks/[id]/route.ts`, `app/api/tasks/[id]/complete/route.ts`), which is consistent with the breaking change introduced in Next.js 15 where params became asynchronous.

### 1.2 Server Component Default

API routes are pure server code. No client-side data fetching patterns (SWR, React Query) are visible in the API layer. The component layer correctly uses `"use client"` only where interactivity is required.

### 1.3 Gaps Against App Router Best Practices

**Route Handler response type inconsistency.** Cron routes and push routes use `NextResponse.json(...)` from `next/server`, while all other routes use the native `Response.json(...)`. Both work identically in the App Router runtime, but the inconsistency signals that the cron and push routes were authored separately without adopting the project-wide convention. This should be normalised.

**No Server Actions.** All mutations go through API routes. This is a deliberate and defensible choice (explicit HTTP semantics, easier to rate-limit, easier to document), but it means every mutation requires a full client-side `fetch()`. For forms and simple state transitions, Server Actions would reduce round-trips. This is an observation, not a deficiency — the current approach is architecturally valid for an app targeting PWA/mobile where API contracts are meaningful.

**No streaming or Suspense boundaries.** The dashboard and task list pages load data synchronously in Server Components. For the current data volumes this is acceptable. As the task and completion history grows, adding Suspense-based streaming for stats would improve perceived performance.

## 2. Separation of Concerns

### 2.1 lib/ vs routes vs components

The discipline is strong:

- API route handlers are consistently thin: authenticate, validate, call one lib function, return response.
- No SQL queries appear in route files.
- No business logic appears in components.
- The `lib/validators/` boundary is consistently respected — all external input is validated before reaching `lib/` functions.

### 2.2 Coupling Violations

**`lib/tasks.ts` imports `lib/gamification.ts`.** The `completeTask` function in `lib/tasks.ts` directly imports and calls `updateStreak`, `checkAndUnlockAchievements`, and `getLevelForCoins` from `lib/gamification.ts`. This is a coupling that makes `completeTask` responsible for orchestrating gamification side-effects, rather than the route or a dedicated service.

The consequence is that:
- `lib/gamification.ts` cannot be tested independently without triggering task logic.
- Any change to the gamification contract (e.g., adding XP separate from coins) requires editing `lib/tasks.ts`.
- The `completeTask` function has grown to 94 lines and handles six distinct concerns.

**Architectural recommendation:** Introduce an application-layer orchestrator (e.g., `lib/actions/complete-task.ts`) that calls `lib/tasks.ts` for the task mutation and `lib/gamification.ts` for rewards. The `lib/tasks.ts` function would do only the task update and completion-log insert. This mirrors the standard service/use-case pattern and aligns with CLAUDE.md's intent that business logic belongs in `lib/` but that `lib/` functions should remain focused.

**`lib/daily-quest.ts` contains dashboard stats.** The `getUserStats` function at the bottom of `lib/daily-quest.ts` (lines 439–459) has no relationship to daily quest logic. It queries `users.coins`, `users.streakCurrent`, and `users.level`. It belongs either in `lib/users.ts` or `lib/gamification.ts`. As written, a developer looking for "get user stats" has no obvious place to find it unless they know to look in the daily quest module.

**`lib/push.ts` contains domain queries.** The `sendStreakReminders` and `sendDailyQuestNotifications` functions in `lib/push.ts` embed their own DB queries to determine which users are eligible. This means the push module owns user-eligibility logic that properly belongs in the user or notification domain. A cleaner boundary would have a `lib/notifications.ts` module that answers "which users need a streak reminder?" and passes that list to `lib/push.ts` for the actual send.

## 3. Data Model Soundness

### 3.1 Schema Overview

The 10-table schema is well-normalised for the problem domain:

```
users (1) ──< accounts (N)            OAuth provider links
users (1) ──< sessions (N)            Active sessions
users (1) ──< topics (N)              Project buckets
users (1) ──< tasks (N)               Core entities
topics (1) ──< tasks (N)              Optional grouping
tasks (1) ──< task_completions (N)    Completion log
users (1) ──< wishlist_items (N)      Wishlist
achievements (1) ──< user_achievements (N)   Earned achievements
users (1) ──< user_achievements (N)
```

Foreign key constraints use `ON DELETE CASCADE` for user-owned data, which is correct — deleting a user removes all their data. The `tasks.topicId` correctly uses `ON DELETE SET NULL` so tasks survive topic deletion.

### 3.2 Denormalized Gamification Columns

`users.coins`, `users.level`, `users.streakCurrent`, `users.streakMax`, and `users.streakLastDate` are denormalized summaries. They are derived from `task_completions` but maintained by application code rather than DB triggers or views. This is a deliberate trade-off for read performance (single-row user fetch gives full gamification state) but creates the integrity risk identified in L1 B-01/B-02/B-03.

The fundamental architectural flaw is that these summary fields can diverge from the completion log because they are updated outside any transaction that also writes the completion log. The correct fix is not just wrapping in a transaction (though that is necessary) but also periodically verifying or recomputing summaries from the canonical `task_completions` source to detect drift. This is a pattern known as read-model consistency and is a standard concern for any CQRS-adjacent design.

**Recommendation:** Add a `POST /api/admin/recalculate-user-stats` endpoint (admin-only) that recomputes coins, level, and streak from the completion log for a given user. This serves as both a repair tool and an auditing mechanism.

### 3.3 Missing Constraint: `user_achievements` Uniqueness

The `user_achievements` table has no unique constraint on `(userId, achievementId)`. The application prevents duplicates in code, but a concurrent-completion race (L1 H-06) can bypass this. A `UNIQUE (user_id, achievement_id)` constraint should be declared in the schema and enforced at the DB level. This is a data model deficiency, not just a code deficiency.

### 3.4 `is_daily_quest` as Mutable Boolean Flag

Storing `isDailyQuest` as a boolean column on the `tasks` table is a design choice that creates implicit state: at most one task per user should have `isDailyQuest = true` at any given time, but there is no DB-level constraint enforcing this. The schema relies entirely on application logic to maintain this invariant.

A more robust design would store the daily quest assignment in a separate table:

```sql
daily_quest_assignments (
  id          uuid PK,
  user_id     uuid FK users,
  task_id     uuid FK tasks,
  date        date NOT NULL,
  UNIQUE (user_id, date)
)
```

This separates the "which task is today's quest" concern from the task entity, makes historical quest data queryable, and uses the DB to enforce the one-quest-per-user-per-day invariant. It also eliminates the `selectDailyQuest` cleanup step that clears stale `isDailyQuest` flags.

This is a structural recommendation, not required for the current phase, but worth recording as a future schema migration target.

### 3.5 `nextDueDate` Computed by Application Code

For `RECURRING` tasks, `nextDueDate` is computed in `completeTask` using `new Date()` plus the interval. This is fine for now but has two risks:
- If the server clock drifts or the TZ setting is wrong, the next due date will be calculated incorrectly (directly related to H-03 from L1).
- The formula is embedded in `completeTask` only. If a recurring task is created with a `dueDate`, the initial `nextDueDate` is never populated — it is null until the first completion. This means a newly-created recurring task will never appear in the "due recurring task" query in `selectDailyQuest` (priority 3) until after its first completion.

**Recommendation:** Populate `nextDueDate` on task creation for `RECURRING` tasks (set to `dueDate` if provided, otherwise `today + recurrenceInterval`).

### 3.6 `price` and `monthlyBudget` as `decimal`

Using `decimal(10, 2)` for monetary values is correct. Drizzle returns these as strings to preserve precision. The application code should not coerce these to JavaScript floats for arithmetic — this is a potential precision loss point in `getBudgetSummary` in `lib/wishlist.ts` if `parseFloat()` is used. This should be confirmed and, if necessary, replaced with a decimal arithmetic library.

## 4. API Design Consistency

### 4.1 Response Envelope

All API routes return `{ data }` or `{ error: string }` envelopes consistently. Named properties are used for the data payload (`{ task }`, `{ tasks }`, `{ quest }`, `{ items }`) rather than a generic `{ data }` wrapper. This is slightly inconsistent with REST conventions but is a deliberate legibility choice and is applied uniformly.

### 4.2 HTTP Status Codes

Status code usage is consistent:
- `200` for successful reads
- `201` for successful creates
- `400` for JSON parse failures
- `401` for unauthenticated
- `404` for not-found (scoped by userId, so this is also an implicit ownership check)
- `409` for conflict (already completed, cannot uncomplete)
- `422` for validation failures with field-level error details
- `429` for rate limiting with `Retry-After`
- `500` for unexpected errors

The only gap: `DELETE /api/tasks/:id` returns `{ success: true }` on a 200 rather than `204 No Content`. While `204` is more idiomatic for deletions, this is a minor inconsistency.

### 4.3 Rate Limiting Coverage

Rate limiting is applied on:
- `POST /api/tasks` — create task
- `POST /api/tasks/:id/complete` — complete task
- `POST /api/wishlist` — create wishlist item
- `POST /api/daily-quest/postpone` — postpone quest

Not rate-limited:
- `POST /api/daily-quest` (force-reselect) — no limit, see L1 N-08
- `PATCH /api/tasks/:id` — update task
- `DELETE /api/tasks/:id` — delete task
- `POST /api/topics` — create topic
- `PATCH /api/topics/:id` — update topic
- `DELETE /api/topics/:id` — delete topic
- `POST /api/push/subscribe` — register push subscription

The absence of rate limiting on topic and task mutations (PATCH, DELETE) is acceptable for the current scale. Topic creation deserves a rate limit. The push subscribe endpoint should have one to prevent subscription churn abuse.

### 4.4 Error Message Leakage

In `app/api/tasks/[id]/complete/route.ts` (line 53):

```typescript
const message = error instanceof Error ? error.message : "Failed to complete task";
```

The raw `Error.message` from `lib/tasks.ts` is returned directly to the client. The messages like "Task not found or access denied" and "Task is already completed" are intentional user-facing messages, but any unexpected DB error message would also be exposed. The pattern used in `app/api/tasks/route.ts` (lines 59–62) is safer — it logs the error and returns a generic message. The complete/uncomplete route should adopt the same pattern for the 500 path.

## 5. Rate Limiter Scalability

This section expands on L1 N-07 with full architectural context.

### 5.1 Current Design

`lib/rate-limit.ts` is a fixed-window in-memory rate limiter backed by a `Map<string, RateLimitEntry>`. The code itself acknowledges the multi-replica limitation. The architectural issues are:

**Memory leak.** Expired entries are never evicted. Each unique user action key occupies memory permanently. With 1,000 active users each triggering 5 rate-limit buckets, the Map accumulates 5,000 entries that are logically expired but never freed.

**Per-replica effective limit.** With `replicas: 2` in the Kubernetes manifest, a determined user can achieve 2x the nominal limit by routing requests to both pods. This is not a theoretical concern — load balancers with round-robin routing will naturally distribute requests.

**Process restart resets all limits.** A rolling deployment or pod restart wipes all rate limit state. A user hitting the limit just before a deployment gets a free reset.

### 5.2 Architectural Options

| Option | Complexity | Effectiveness | Cost |
|---|---|---|---|
| Current in-memory | Minimal | Only works for single pod | Zero infra |
| Sticky sessions (consistent hashing) | Low | Works for single-user consistency | Load balancer config |
| Redis sliding window | Medium | True distributed enforcement | Redis instance |
| Database-backed (Postgres) | Medium | True distributed, no new dep | Query overhead |
| Upstash / Vercel KV | Low | Managed, ephemeral | SaaS cost |

For a self-hosted personal app on Kubernetes, the pragmatic choice is a Redis deployment (one additional StatefulSet) or Upstash if running on a cloud platform. Given the existing Postgres deployment, a DB-backed rate limiter (a `rate_limit_events` table with TTL-indexed timestamps) is an option that avoids a new dependency.

**Recommendation for next phase:** The in-memory limiter is acceptable for the initial single-user use case. Before opening Momo to multiple users, the rate limiter must be replaced. Document this explicitly as a phase prerequisite.

## 6. `completeTask` Atomicity — Architectural Framing

The L1 review identified B-01, B-02, and B-03 as blocking bugs. From an architecture perspective, these are symptoms of a deeper structural issue: the domain operation "complete a task" spans six tables and must be atomic, but the architecture provides no transaction boundary.

### 6.1 Current Control Flow

```
completeTask(taskId, userId)
  getTaskById()                    -- SELECT tasks
  db.update(tasks)                 -- UPDATE tasks (set completedAt or nextDueDate)
  db.insert(taskCompletions)       -- INSERT task_completions
  incrementUserCoins()             -- SELECT users; UPDATE users.coins
  getUserLevel() + getLevelForCoins()
  db.update(users)                 -- UPDATE users.level (conditional)
  updateStreak()                   -- SELECT users; UPDATE users.streak*
  db.select(taskCompletions)       -- SELECT count for achievements
  checkAndUnlockAchievements()     -- SELECT achievements; SELECT user_achievements; INSERT user_achievements
```

This is nine separate DB operations across six tables with no transaction. Any failure after the task update but before the achievement insert leaves the system in an inconsistent state.

### 6.2 Transaction Boundary

The entire sequence from step 2 (UPDATE tasks) through step 9 (INSERT user_achievements) must execute inside a single Drizzle transaction. The `db.transaction(async (tx) => { ... })` API is available in Drizzle and all query calls within the lambda should use the `tx` handle.

The `incrementUserCoins` helper must be replaced with an atomic SQL update:

```typescript
// Atomic increment — eliminates the read-modify-write race
.set({ coins: sql`${users.coins} + ${coinsEarned}` })
```

### 6.3 Idempotency

`completeTask` has no idempotency key. A double-tap (two identical POST requests within milliseconds) can result in two completions being recorded if both requests pass the "already completed" check before either write completes. Wrapping in a transaction reduces the window but does not eliminate it for non-serializable isolation levels.

A complete fix requires either:
- A database-level unique constraint on `task_completions(task_id, date_trunc('day', completed_at))` for non-recurring tasks (enforce one completion per task per day), or
- An idempotency key in the request (client-generated nonce) that is stored and checked.

This is an architectural recommendation for phase hardening, not an immediate blocker for a single-user app.

## 7. Timezone Architecture

### 7.1 Current State

The application uses two incompatible strategies for generating "today":

**Strategy A — Local time (correct for streak/quest logic):**
```typescript
// lib/gamification.ts and lib/daily-quest.ts
function getTodayString(): string {
  const now = new Date();
  return `${now.getFullYear()}-${...}-${...}`;
}
```

**Strategy B — UTC (used in two specific computations):**
```typescript
// lib/tasks.ts completeTask — recurring next due date
const nextDueStr = nextDue.toISOString().split("T")[0];

// lib/daily-quest.ts postponeDailyQuest — tomorrow
const tomorrowStr = tomorrow.toISOString().split("T")[0];

// lib/push.ts sendStreakReminders — today start
todayStart.setUTCHours(0, 0, 0, 0);
```

Strategy B will produce a date that is one day earlier than intended for any user in a UTC-positive timezone (e.g., UTC+2) when their local clock is between 00:00 and 02:00.

### 7.2 Architectural Decision Required

There are two valid approaches, but the team must pick one and enforce it consistently:

**Option 1 — Server-UTC convention.** Set `TZ=UTC` in all deployment environments (Docker, Kubernetes, local dev). All date computations use UTC methods. All `date` column values are UTC dates. Document in `docs/deployment.md` and `.env.example` that `TZ=UTC` is a required deployment invariant.

**Option 2 — User timezone awareness.** Store a `timezone` column on users (e.g., `"Europe/Vienna"`). Pass the user timezone into every date computation. This is architecturally cleaner for a multi-timezone user base but significantly more complex to implement correctly.

For a personal task management app where the server and user are in the same timezone, Option 1 is the right call. The `TZ=UTC` environment variable must be added to:
- The Kubernetes deployment manifest (`env:` section)
- The Docker run instructions in `docs/deployment.md`
- `.env.example` as a documentation note
- The `lib/env.ts` runtime check (assert that `process.env.TZ === "UTC"` or log a warning in non-test environments)

### 7.3 Shared `getTodayString` Utility

Both `lib/gamification.ts` and `lib/daily-quest.ts` define identical private `getTodayString()` functions (L1 NIT-01). This duplication means that when the function is corrected to use UTC methods, it must be corrected in two places. A shared `lib/utils/date.ts` module with an exported `getTodayString()` is a prerequisite for any timezone fix.

## 8. Push Notification Architecture

### 8.1 Fan-out Design

The `sendStreakReminders` and `sendDailyQuestNotifications` functions fan out push notifications to all eligible users by iterating a JavaScript loop. This is architecturally equivalent to a serial, single-threaded fan-out. For a small user count (< 100) this is acceptable. Beyond that, the cron job will take longer than one minute to complete and will be re-triggered before finishing.

The architectural options are:
- Parallel fan-out with `Promise.all` or `Promise.allSettled` (increases DB connection pressure but reduces wall-clock time)
- Batch processing with a queue (correct solution for production scale, requires a queue service)
- Limiting to a `WHERE notification_time = current_time` filter so each cron tick handles only users whose preferred notification time matches now (reduces per-tick volume)

For the current single-user use case, the serial fan-out is fine. The `notificationTime` column on users already provides the data needed for per-user scheduling — it is not being used for this purpose currently.

### 8.2 `checkCompletedToday` Double-Query Pattern

The L1 review (B-04) identified that `sendStreakReminders` makes two rounds of queries per user to determine if they completed a task today, with neither query filtering by date at the DB level. The `todayStart` variable computed at the start of the function is passed to `checkCompletedToday` but never included in any Drizzle `where()` clause. The date filter happens entirely in JavaScript via `.some((r) => new Date(r.completedAt) >= todayStart)` after loading all completion rows.

From an architecture standpoint, this is the same pattern as L1 H-04 (`getUserTopics` loading all tasks to count in JS). The database should filter; the application should not. The `gt(taskCompletions.completedAt, todayStart)` Drizzle operator exists precisely for this case.

### 8.3 VAPID Key Access Bypasses `clientEnv`

`lib/push.ts` accesses `process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY` directly (L1 NIT-02) instead of through `clientEnv`. The project rule requires all env var access to go through `lib/env.ts`. This is a layering violation that also bypasses the validation wrapper — if the key is missing, `push.ts` will proceed silently (via the `isVapidConfigured()` no-op path) rather than failing at startup with a clear error message.

## 9. Deployment Architecture

### 9.1 Kubernetes Manifest Quality

The manifests are production-quality for a small self-hosted workload:
- Non-root user (UID 1001) and `allowPrivilegeEscalation: false`
- Pod anti-affinity to spread replicas across nodes
- Liveness and readiness probes against `/api/health`
- Resource requests and limits set
- Secrets injected via `secretRef` rather than plain `env:`

### 9.2 Two-Replica Topology vs. Single-Process Assumptions

The deployment targets 2 replicas. As discussed in section 5, this breaks:
- The in-memory rate limiter (different pods have independent Maps)
- The `isDailyQuest` boolean flag (not a concurrency issue per se, but two pods can both run `selectDailyQuest` for the same user simultaneously, resulting in two tasks being assigned)

The `selectDailyQuest` function checks for an existing quest, clears completed quests, then selects a new one — all outside a transaction. Two concurrent GET requests to `/api/daily-quest` for the same user can both pass the "no existing quest" check and assign different tasks as the daily quest. The second write wins, and the first-assigned task is left with `isDailyQuest = true` until the next cleanup pass.

This is a distributed systems correctness issue. The mitigation options are:
- Advisory locking per user at the DB level (`SELECT ... FOR UPDATE` on a user row)
- Serializing daily quest selection through a queue or leader-election mechanism
- Accepting last-write-wins semantics and documenting it as a known limitation (acceptable for a personal app)

### 9.3 Postgres StatefulSet Caveats

The `postgres-statefulset.yaml` is correctly labelled as single-replica and not suitable for HA. The `ReadWriteOnce` PVC access mode is correct for a single-replica StatefulSet. The inline comment correctly recommends CloudNativePG or similar for HA. No issues.

### 9.4 CI/CD Pipeline

The `build-and-publish.yml` workflow:
- Builds on every push to `main` and every semver tag
- Pushes to GHCR, Docker Hub, and Quay.io simultaneously
- Uses `docker/metadata-action` for consistent tagging (branch, semver, SHA prefix)
- Uses GitHub Actions layer cache

The pipeline has no test or lint step. There is no `npm test` or `npm run lint` job. Any TypeScript error or ESLint violation merged to main will be shipped in the next image without warning.

**Recommendation:** Add a `ci.yml` workflow that runs on every pull request (or push to main since the project has no feature branches) with:

```yaml
- run: npm ci
- run: npm run lint
- run: npm run typecheck   # tsc --noEmit
- run: npm run test        # if/when tests are added
```

Make the build-and-publish job depend on the CI job passing. This enforces the TypeScript strictness mandated in CLAUDE.md at the pipeline level.

## 10. Security Architecture

### 10.1 Authentication Boundary

Every API route that reads or mutates user data calls `auth()` as its first statement and returns `401` if no session. The pattern is consistent. This is the correct boundary placement.

**Gap:** The `POST /api/admin/seed` route (L1 B-05) only checks that a session exists. Any authenticated user can seed achievements. The route must gate on an admin condition before it can be considered correctly positioned within the security architecture.

### 10.2 Authorization Model

The authorization model is ownership-based: every DB query includes `eq(table.userId, session.user.id)`. There is no role-based access control, which is appropriate for a single-user-per-account model. The model is consistently applied across all `lib/` functions.

**Partial gap:** `assignDailyQuest` in `lib/daily-quest.ts` re-fetches the task after update without a userId filter (L1 N-06). The update itself is correctly scoped, so there is no privilege escalation vector in practice, but the re-fetch does not match the project's stated defense-in-depth standard.

### 10.3 Cron Endpoint Protection

The `CRON_SECRET` guard uses an opt-in pattern: the secret is only checked if it is configured. When `CRON_SECRET` is absent, the endpoint accepts any request (L1 H-02). This inverts the expected security posture — the default should be "deny unless configured", not "allow unless configured".

This is an architectural decision in `lib/env.ts` where `CRON_SECRET` is typed as `optional`. Making it required at the schema level would force the operator to consciously set it, but would break zero-config local development. The resolution is to:
- Keep it optional in `env.ts`
- Change the route guard from `if (cronSecret) { check }` to `if (!cronSecret || token !== cronSecret) { return 401 }`

This ensures that the endpoint is protected by a fixed "no auth configured" 401 when no secret is set, rather than being open.

### 10.4 Input Validation Completeness

Zod schemas cover all mutation endpoints. The `dueDate` field accepts any string matching `YYYY-MM-DD` format but does not validate whether the date is a valid calendar date (e.g., `2026-02-30` would pass). This is a minor gap — the Postgres `date` type will reject an invalid date at the DB level, but the error will surface as a 500 rather than a 422. Adding a Zod refinement with `new Date(val).getMonth() === parseInt(val.split('-')[1]) - 1` or using `z.coerce.date()` would return a user-friendly error.

## 12. Scalability Implications Summary

| Concern | Current State | Trigger Point | Recommended Fix |
|---|---|---|---|
| In-memory rate limiter | Per-pod, no eviction | > 1 pod OR > ~10k users | Redis or DB-backed rate limiter |
| `getUserTopics` full task scan | O(N) tasks per request | > 500 tasks per user | SQL GROUP BY aggregate |
| `sendStreakReminders` full completion scan | O(N) completions per user | > 200 completions per user | Add `gte(completedAt, todayStart)` filter |
| Serial push fan-out | Single-threaded loop | > 100 subscribed users | Promise.allSettled or queue |
| `selectDailyQuest` non-atomic | TOCTOU race | > 1 replica | DB transaction or advisory lock |
| `completeTask` non-transactional | Data inconsistency | > 1 concurrent session per user | Drizzle transaction |
| `task_completions` unbounded growth | No archival strategy | > 10,000 completions per user | Partition or archive old records |
