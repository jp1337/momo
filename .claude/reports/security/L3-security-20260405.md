## Findings

### HIGH — `createTask` RECURRING `nextDueDate` still uses server UTC clock

**Location:** `lib/tasks.ts:148`
**Category:** Business Logic / Data Integrity (CWE-840)
**Status:** Confirmed carry-over from previous review — NOT fixed in this delta

**Description:**
When a new RECURRING task is created without an explicit `dueDate`, the initial `nextDueDate` defaults to the server's UTC date:

```ts
nextDueDate: input.type === "RECURRING"
  ? (input.dueDate ?? new Date().toISOString().split("T")[0])
  : null,
```

`createTask` receives no `timezone` parameter and the `POST /api/tasks` route does not read or forward any timezone from the request body. A user in UTC-8 creating a task at 23:00 local time will receive a `nextDueDate` of UTC tomorrow (their local today + 1 day), causing the daily quest algorithm to skip the task until the following day.

This is the same HIGH finding identified in the previous review. The `completeTask` path was fixed (timezone now threaded correctly via `TimezoneSchema`), but `createTask` remains unaddressed.

**Recommendation:**
1. Add `timezone?: string | null` to `CreateTaskInputSchema` in `lib/validators/index.ts` (reuse `TimezoneSchema`).
2. Add `timezone?: string | null` parameter to `createTask(userId, input, timezone?)`.
3. Replace `new Date().toISOString().split("T")[0]` with `getLocalDateString(timezone)`.
4. The `POST /api/tasks` route already passes the full Zod-parsed body to `createTask` — no route changes required once the schema is extended.

**References:** CWE-840 (Business Rule Bypass), prior finding `lib/tasks.ts:148` in L3-security-20260404.md

### MEDIUM — `selectDailyQuest` timezone anchoring regressed to server UTC for "completed today" guard

**Location:** `lib/daily-quest.ts:234`
**Category:** Business Logic / Authorization Bypass (CWE-840)
**Status:** Partial fix introduced — new issue created

**Description:**
In `selectDailyQuest`, the "already completed a quest today" guard now computes `todayStart` as:

```ts
const today = getLocalDateString();          // no timezone argument
const todayStart = new Date(`${today}T00:00:00Z`);
```

`getLocalDateString()` is called without a timezone argument, so it falls back to the server's local clock (UTC in production containers). The guard will incorrectly allow or deny a quest based on UTC midnight rather than the user's midnight.

The fix correctly changed `T00:00:00` to `T00:00:00Z` (preventing incorrect local-time parsing), but the `getLocalDateString()` call still needs a timezone parameter. `selectDailyQuest` does not currently accept a timezone argument.

This was flagged as NON-BLOCKING in the L2 context (`lib/daily-quest.ts:234`) and remains unresolved.

**Recommendation:**
Add `timezone?: string | null` parameter to `selectDailyQuest(userId, timezone?)` and thread it through to both `getLocalDateString(timezone)` calls within the function body (lines ~234 and ~319). The `GET /api/daily-quest/route.ts` caller should pass the client's timezone from a request header or body — same pattern already used by the postpone route.

**References:** CWE-840; prior L2 finding `lib/daily-quest.ts:234`

### MEDIUM — `breakdownTask` does not increment `totalTasksCreated` counter

**Location:** `lib/tasks.ts:562–574`
**Category:** Data Integrity / Counter Desynchronisation (CWE-840)
**Status:** Confirmed carry-over from previous review — NOT fixed in this delta

**Description:**
`breakdownTask` calls `tx.insert(tasks).values(subtaskTitles.map(...))` to create N new subtask rows, but never updates `users.totalTasksCreated`. The `createTask` path now correctly increments the counter inside a transaction, but `breakdownTask` bypasses that path, silently under-counting created tasks.

This means the statistics page underreports `totalTasksCreated` for any user who has used the breakdown feature. No security boundary is violated, but the integrity of the statistic is compromised.

**Recommendation:**
After the bulk insert inside `breakdownTask`'s transaction, add:

```ts
await tx
  .update(users)
  .set({ totalTasksCreated: sql`${users.totalTasksCreated} + ${subtaskTitles.length}` })
  .where(eq(users.id, userId));
```

**References:** CWE-840; prior L2 finding `lib/tasks.ts:527`

### LOW — `updateStreak` moved outside transaction — partial-success window on failure

**Location:** `lib/tasks.ts:271–281` (post-transaction streak update)
**Category:** Data Integrity / TOCTOU (CWE-362)
**Status:** New finding introduced in this delta

**Description:**
The delta deliberately moved `updateStreak` outside the main `completeTask` transaction with the comment "a transient streak failure should not roll back the task completion or coin award." This is a valid design choice for resilience, but it introduces a small partial-success window:

- Task is marked complete and coins are awarded (inside transaction — atomic).
- `updateStreak` fails (network blip, DB overload).
- The catch block logs the error and continues.
- **Result:** The user's streak is NOT updated even though a task was completed. On the next call, `updateStreak` may or may not recover the streak depending on whether it reads `last_active_date`.

If `updateStreak` uses `last_active_date` as its idempotency anchor, a missed update causes a permanent streak reset that cannot be recovered later (since the day's timestamp is gone). The user loses a streak day through no fault of their own.

**Recommendation:**
Confirm that `updateStreak` is idempotent when called again for the same day (i.e., it reads `last_active_date` and only advances if today is the next eligible date). If so, the current design is acceptable. If not, either log a structured alert (with `userId`) so a human can investigate, or persist a "streak needs recalculation" flag for deferred repair.

**References:** CWE-362

### LOW — `dispatchCoinsEarned` uses `window` directly — server-side render crash risk

**Location:** `lib/client/coin-events.ts:20–23`
**Category:** Runtime Error / Environment Guard (CWE-391)
**Status:** New finding

**Description:**
`dispatchCoinsEarned` unconditionally accesses `window`:

```ts
window.dispatchEvent(new CustomEvent(COINS_EARNED_EVENT, { detail: { delta } }));
```

The function is correctly placed under `lib/client/` and documented as client-safe, but Next.js Server Components and edge runtimes can accidentally import client modules, causing a `ReferenceError: window is not defined` at runtime. The existing guard (`typeof delta !== "number" || ...`) does not protect against this.

There is no current evidence this module is imported server-side, so this is LOW risk, but it is a latent defect.

**Recommendation:**
Add a `typeof window === "undefined"` guard at the top of the function body:

```ts
if (typeof window === "undefined") return;
```

This makes the function safe to call in any rendering context without any behavioural change on the client.

**References:** CWE-391

### INFO — `TimezoneSchema` defined inline in postpone route, not replaced with shared import

**Location:** `app/api/daily-quest/postpone/route.ts:19`
**Category:** Code Quality / DRY violation

**Description:**
`PostponeBodySchema` still defines `timezone: z.string().max(64).optional().nullable()` inline rather than importing and composing `TimezoneSchema` from `lib/validators/index.ts`. This was identified as a NON-BLOCKING finding in the L2 context. The `TimezoneSchema` export was added to `lib/validators/index.ts` in this delta specifically to address this, but the postpone route was not updated to use it.

This is a consistency issue: if the max-length limit changes on `TimezoneSchema`, the inline definition will drift.

**Recommendation:**
Replace the inline definition with:

```ts
import { TimezoneSchema } from "@/lib/validators";
// ...
const PostponeBodySchema = z.object({
  taskId: z.string().uuid("taskId must be a valid UUID"),
  timezone: TimezoneSchema,
});
```

### INFO — Migration SQL runs `UPDATE users SET ... COUNT(*)` without index on `tasks.user_id`

**Location:** `drizzle/0004_messy_zodiak.sql`
**Category:** Performance / Availability (not a security finding)

**Description:**
The backfill `UPDATE "users" SET "total_tasks_created" = (SELECT COUNT(*) FROM "tasks" WHERE "tasks"."user_id" = "users"."id")` performs a correlated subquery against `tasks` for every user row. At scale, this is a full-table-scan pattern that can hold locks for an extended period. On a live database with many users and tasks, this migration could cause service degradation.

This is informational for this codebase at its current scale.

### INFO — `scripts/migrate.mjs` `statement_timeout = 30000` applies to catalog queries, not the migration itself

**Location:** `scripts/migrate.mjs:228`
**Category:** Operational

**Description:**
`SET statement_timeout = 30000` applies to the inspection client connection used for catalog queries. The Drizzle `migrate()` call uses a separate connection pool, so the timeout does not cover the actual DDL/DML migration statements. Long-running migrations (like the backfill above) are not protected by this timeout.

No security implication; noted for operational awareness.
