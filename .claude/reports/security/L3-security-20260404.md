### HIGH — Missing Input Length Cap on `timezone` in Complete Route

**Location:** `app/api/tasks/[id]/complete/route.ts` lines 42–48
**Category:** OWASP A04 – Insecure Design / A03 – Injection-adjacent DoS
**Status:** BLOCKING (carried forward from L1/L2 pre-scan context)

**Description:**
The POST `/api/tasks/:id/complete` route accepts a `timezone` field from the request body with only a `typeof === "string"` guard:

```typescript
if (typeof body?.timezone === "string") timezone = body.timezone;
```

This string — with no maximum length constraint — is passed directly to `completeTask()`, then to `updateStreak()` in `lib/gamification.ts`, and finally to `getLocalDateString()` / `getLocalYesterdayString()` in `lib/date-utils.ts`, where it is forwarded to `new Intl.DateTimeFormat("en-CA", { timeZone: timezone })`.

By contrast, the `/api/daily-quest/postpone` route correctly validates the same field with `z.string().max(64)` via Zod before passing it downstream. The inconsistency means an authenticated user can send an arbitrarily long string (e.g. 1 MB) on every task completion request. The V8/Node.js `Intl.DateTimeFormat` constructor must parse the string before it can reject it as invalid. Under sustained concurrent requests this creates a CPU and memory pressure vector. The garbage collected try/catch in `date-utils.ts` only fires after the damage is done.

The string is never persisted to the database and does not reach SQL, so SQL injection is not a concern here. XSS is also not a risk: the value is processed server-side only and never reflected in a response.

**Recommendation:**
Validate the `timezone` field in the complete route using the same Zod schema pattern already used in the postpone route. A `z.string().max(64).optional().nullable()` constraint would eliminate the oversized-string vector. Ideally, extract a shared `TimezoneSchema` constant in `lib/validators/index.ts` and import it in both routes to enforce the constraint from a single source of truth.

**References:** CWE-400 (Uncontrolled Resource Consumption), CWE-20 (Improper Input Validation)

### HIGH — Split-Brain Timezone in Quest Selection vs. Postpone/Complete

**Location:** `lib/daily-quest.ts` lines 43–49 (`getTodayString`), lines 149, 195, 246–247
**Category:** OWASP A04 – Insecure Design
**Status:** BLOCKING (carried forward from L1/L2 pre-scan context)

**Description:**
The `selectDailyQuest()` and `forceSelectDailyQuest()` functions call a module-local `getTodayString()` helper (line 43) that always uses the server's local time (UTC in production containers). This string is used in the overdue-task query (`lt(tasks.dueDate, today)` — line 165), the recurring-task query (`lte(tasks.nextDueDate, today)` — line 195), and the `completedAt >= todayStart` check (line 248) that determines whether a new quest should be selected.

The `postponeDailyQuest()` function and `completeTask()` downstream correctly call `getLocalDateString(timezone)` from `lib/date-utils.ts` for all date comparisons. This means the date used to select a quest (UTC) can differ from the date used to postpone or complete it (user local time) for users in UTC+ timezones late in the day.

**Concrete manipulation scenario:**
A user in UTC+12 at 23:45 local time (11:45 UTC the same day) postpones their quest, which writes `questPostponedDate = "2026-04-04"` (local). `selectDailyQuest()` then re-evaluates using UTC date `"2026-04-03"`, treats the quest as not-yet-selected-today, and immediately assigns a new quest for what it believes is yesterday. The user can then postpone again. This allows circumventing the daily postpone limit by one additional postpone per day for users in UTC+ timezones.

This is an application logic security issue: it does not leak data or allow privilege escalation, but it allows manipulation of a business-rule boundary (the postpone budget) in a predictable and repeatable way.

**Recommendation:**
`selectDailyQuest()` and `forceSelectDailyQuest()` must accept an optional `timezone` parameter and pass it to `getLocalDateString()` from `lib/date-utils.ts` instead of using the local `getTodayString()` helper. The local helper should be removed. The calling API routes (`/api/daily-quest/route.ts`) should read the timezone from the same request body field already parsed for postpone. Alternatively, persist the user's preferred timezone in the `users` table and read it from the DB in all quest functions, which would also eliminate the trust issue of accepting a client-supplied timezone entirely.

**References:** CWE-840 (Business Rule Bypass), CWE-697 (Incorrect Comparison)

### MEDIUM — `migrate.mjs` Reads Migration SQL Files from Disk at Container Startup with Full DB Credentials

**Location:** `scripts/migrate.mjs` lines 210–258
**Category:** OWASP A05 – Security Misconfiguration
**Status:** New finding

**Description:**
The migration script runs at container startup with a connection pool authenticated by `DATABASE_URL` (which holds the full production DB credentials). Beyond calling `migrate()`, the script now directly executes five categories of additional queries: `isMigrationTracked`, `isMigrationAppliedInDb` (which calls `tableExists`, `indexExists`, `columnExists`), `seedOneMigration`, and `DELETE FROM drizzle.__drizzle_migrations`.

The attack surface concern is not in the query parameterization — all five helpers use parameterized `$1` placeholders correctly and are therefore safe from SQL injection. The concern is the combination of:

1. The script reads `.sql` migration files from disk (`readFileSync`) and then parses their content with client-side regex (`parseCreatedTables`, `parseCreatedIndexes`, `parseAlterAddColumns`) before deciding whether to seed tracking records. If the `drizzle/` folder on the container image were tampered with (supply-chain or image-layer attack), the script would process attacker-controlled SQL filenames and table/column names. These strings are used only as parameterized query values, so direct injection is blocked, but a crafted migration filename or table name could produce a misleading tracking state.

2. The `DELETE FROM drizzle."__drizzle_migrations"` at line 230–233 removes tracking records based on the result of client-side file parsing. If `isMigrationAppliedInDb` returns false due to a parsing failure (e.g. the regex does not match a legitimately formatted `CREATE TABLE` statement), the script will delete valid tracking entries, causing `migrate()` to re-run already-applied migrations. This is a correctness/availability issue that becomes a security concern if the re-run migration has side effects (e.g. `DROP TABLE IF EXISTS` before re-create).

3. The `parseAlterAddColumns` regex (line 94) is purely additive — it only detects `ADD COLUMN` and never detects `DROP COLUMN`. This means a migration that drops a column will always be considered "not applied in DB" by the column-existence check, causing the script to remove its tracking entry and force migrate() to re-run it.

**Recommendation:**
The parameterized query usage is correct and no SQL injection is present. However, the logic that removes tracking entries based on client-side file parsing should be made more conservative: only delete a tracking entry if both conditions hold *and* the migration SQL file's SHA-256 hash matches the hash stored in the tracking record. This prevents partial parses from triggering destructive resets. Add a guard that skips the delete path entirely for migrations that contain `DROP` statements.

**References:** CWE-494 (Download of Code Without Integrity Check), CWE-693 (Protection Mechanism Failure)

### MEDIUM — `worker-src blob:` Added to CSP Without Explicit Scope Restriction

**Location:** `next.config.ts` line 71
**Category:** OWASP A05 – Security Misconfiguration
**Status:** New finding

**Description:**
The Content-Security-Policy now includes `worker-src 'self' blob:`. The comment in the file correctly attributes this to `canvas-confetti`, which spawns a Web Worker from a `Blob` URL. The addition is technically necessary for that library to function.

The security concern is that `blob:` in `worker-src` permits any JavaScript running in the page context to create a Web Worker from an arbitrary `Blob` containing arbitrary code. If a stored XSS vulnerability were discovered elsewhere in the application, an attacker could use a `new Blob([maliciousCode])` → `URL.createObjectURL()` → `new Worker(blobUrl)` chain to run persistent background code (e.g. crypto-mining, exfiltration, polling) that is harder to detect than inline script execution. The `script-src 'unsafe-inline'` already present in the CSP represents the primary XSS risk; `blob:` in `worker-src` compounds that risk rather than introducing a new independent vector.

The `connect-src 'self'` restriction does limit what such a worker can reach over the network, which partially mitigates the exfiltration risk.

**Recommendation:**
The `blob:` directive cannot be scoped to specific origins — it is all-or-nothing. The mitigation strategy is to eliminate `'unsafe-inline'` from `script-src` in a follow-up hardening task (which would require adopting CSP nonces for Next.js RSC payloads). In the short term, document the `blob:` addition as a known accepted risk in the security runbook. No code change is required for this finding at this time, but it should be tracked as technical security debt.

**References:** CWE-358 (Improperly Implemented Security Check), OWASP CSP Cheat Sheet

### MEDIUM — `coinsEarned` CustomEvent Reads Numeric Value from API Response Without Type Validation

**Location:** `components/tasks/task-list.tsx` lines 274–278; `components/topics/topic-detail-view.tsx` lines 91–95
**Category:** OWASP A03 – Injection (DOM-based)
**Status:** New finding

**Description:**
Both components dispatch a `window` CustomEvent after a successful task completion:

```typescript
window.dispatchEvent(
  new CustomEvent("coinsEarned", { detail: { delta: data.coinsEarned } })
);
```

The `data` object is typed via a local interface `CompleteApiResponse` that marks `coinsEarned` as `number | undefined`. The value originates from a `res.json()` call (line 268 in task-list.tsx) on the API response and is cast with `as CompleteApiResponse`. This cast is not validated — if the API ever returned a non-numeric value for `coinsEarned` (e.g. due to a server-side regression, a proxy injecting content, or a future code change), the numeric assumption would be silently violated.

The `coin-counter` component that listens for this event (`layout/coin-counter.tsx`) applies `delta` to a React state counter. If `delta` were `NaN`, `Infinity`, or a string, the displayed coin total could be corrupted. This is a low-severity DOM manipulation issue, not a code injection vector, because the event is dispatched and consumed entirely within the same origin window context with no cross-origin participant.

The `data.coinsEarned && data.coinsEarned > 0` guard (line 274) prevents dispatch when the value is `undefined` or `0`, but does not guard against strings, `NaN`, or negative values being passed to the event.

**Recommendation:**
Add a `typeof data.coinsEarned === "number" && Number.isFinite(data.coinsEarned)` guard before dispatching the event. This can be done inline in both components. A shared utility function `dispatchCoinDelta(delta: number): void` in a UI utilities module would be the cleanest approach and would ensure the guard is applied consistently across all three dispatch sites (task-list complete, task-list uncomplete, topic-detail-view complete).

**References:** CWE-20 (Improper Input Validation), CWE-116 (Improper Encoding or Escaping of Output)

### LOW — `getLocalTomorrowString` Uses Server-Local `Date` for the Base Date

**Location:** `lib/date-utils.ts` lines 49–69
**Category:** OWASP A04 – Insecure Design
**Status:** New finding

**Description:**
`getLocalTomorrowString()` computes "tomorrow" by calling `new Date()` and incrementing with `setDate(getDate() + 1)`. The `setDate` mutation operates in the server's local time zone (UTC in production). The subsequent `Intl.DateTimeFormat` call then renders this already-shifted date in the user's timezone.

For a user in a UTC- timezone (e.g. UTC-10 at 00:30 local, which is 10:30 UTC), "tomorrow" in the server's perspective is the same calendar day as "today" in the user's perspective. The `+1` is applied to the UTC date before converting to local time, so the result is off by one day for UTC- users when called near midnight. `getLocalDateString()` does not have this problem because it applies `Intl.DateTimeFormat` to the actual current moment without pre-shifting.

This affects `postponeDailyQuest()` in `lib/daily-quest.ts` (line 488), which uses the result to set `due_date = tomorrow` on the postponed task. For UTC- users near midnight, the task may be set due one day further than expected (the day after "local tomorrow"). The impact is usability rather than a direct security vulnerability, but it can interact with the split-brain timezone finding (HIGH #2) to make the postpone limit bypass window larger for UTC- users.

**Recommendation:**
Compute "tomorrow" by constructing a new `Date` representing midnight of the next local day in the user's timezone, rather than UTC-shifting first. The simplest correct approach is to get today's local date string with `getLocalDateString(timezone)`, parse it, add one day, and return it as a formatted string — staying entirely in string space without relying on `Date` arithmetic.

**References:** CWE-697 (Incorrect Comparison)

### LOW — `notes` Field in Task and Topic Creation Has No Maximum Length Validator

**Location:** `lib/validators/index.ts` lines 23–24 (CreateTaskInputSchema), lines 126–128 (CreateTopicInputSchema)
**Category:** OWASP A04 – Insecure Design
**Status:** New finding (observed during validator audit)

**Description:**
The `notes` field in `CreateTaskInputSchema` and `UpdateTaskInputSchema` is validated as `z.string().nullable().optional()` with no maximum length constraint. Similarly, `description` in `CreateTopicInputSchema` and `UpdateTopicInputSchema` has no max length. These strings are stored in the database (PostgreSQL `text` columns). While PostgreSQL `text` has no enforced maximum, the API accepts arbitrarily large payloads into these fields.

An authenticated user could store very large strings (limited only by the request body size limit set by the Next.js/Node.js HTTP server, typically 4 MB by default) in the notes of any task or topic. This consumes database storage per user proportional to the absence of a cap, and large stored notes returned in list queries increase response sizes and memory pressure.

This is not a direct injection vector: Drizzle ORM parameterizes all inserts. The concern is resource exhaustion at the storage and response layer.

**Recommendation:**
Add a reasonable maximum length to `notes` and `description` fields. A limit of 2000–4000 characters is appropriate for a task notes field. For example: `z.string().max(4000).nullable().optional()`.

**References:** CWE-400 (Uncontrolled Resource Consumption)

### INFO — In-Memory Rate Limiter Does Not Persist Across Restarts

**Location:** `lib/rate-limit.ts` lines 20–21
**Category:** OWASP A04 – Insecure Design (design limitation)
**Status:** Informational (pre-existing, no regression)

**Description:**
The rate limiter uses a module-level `Map` that is reset on every process restart and is not shared across multiple replicas. The code itself contains a comment acknowledging this limitation. In a single-replica deployment this is acceptable. However, the comment notes that Kubernetes deployments with `replicas > 1` would allow each pod to enforce limits independently, effectively multiplying the allowed request rate by the number of replicas.

**Recommendation:**

## Cross-Cutting Observations

### Timezone String Trust Boundary

The timezone value originates from `Intl.DateTimeFormat().resolvedOptions().timeZone` on the client (browser). This is a browser-provided IANA timezone name and in normal operation will be a well-formed string like `"Europe/Berlin"`. However, because it is transmitted in an HTTP request body, any client (including a crafted API call via curl or the published OpenAPI spec) can send any string. The `lib/date-utils.ts` functions handle invalid timezone strings by catching the `Intl.DateTimeFormat` constructor exception and falling back to UTC — this is a sound defensive pattern. The remaining risk is purely the unbounded length on the complete route (HIGH finding #1).

### Drizzle ORM Parameterization — Confirmed Safe

All database queries in the reviewed files use Drizzle ORM's query builder (`.select()`, `.update()`, `.insert()`) or parameterized `client.query($1, [value])` calls in `migrate.mjs`. No raw SQL string concatenation of user-supplied values was found. SQL injection risk is absent from this change set.

### XSS via API Response Data — Not Present

The `coinsEarned`, `newLevel`, and `unlockedAchievements` fields from the complete API response are used only to update React component state (numbers, typed objects) and to dispatch a `CustomEvent` with a numeric delta. None of these values are rendered as raw HTML. XSS via the coin event channel is not a realistic attack path in the current code.
