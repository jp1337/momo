### [HIGH] — SEC-01: Admin Seed Endpoint Accessible to All Authenticated Users

**Location:** `app/api/admin/seed/route.ts:19-24`
**Category:** OWASP A01 — Broken Access Control
**CWE:** CWE-285 — Improper Authorization

**Description:**
`POST /api/admin/seed` performs an auth check (`session?.user?.id`) but has no role or privilege check beyond confirming the caller is logged in. Any authenticated user — not just administrators — can trigger `seedAchievements()`. While `seedAchievements()` uses `ON CONFLICT DO NOTHING` and is described as idempotent, this represents a privilege escalation surface: any user can exercise a route explicitly labelled "admin". In a future change, if the seed function is extended or replaced with a more destructive operation, the missing authorization gate will already be missing.

**Recommendation:**
Either remove the endpoint entirely in production (calling `seedAchievements()` at app startup is already documented as the preferred approach), or add an admin role check. At minimum, add an environment gate: check `serverEnv.NODE_ENV === 'production'` and return 403, or introduce an `ADMIN_SECRET` header check analogous to `CRON_SECRET`. No user-facing feature requires this endpoint at runtime.

**References:** CWE-285, OWASP ASVS 4.1.1

### [HIGH] — SEC-02: Cron Endpoints Unprotected When CRON_SECRET Is Not Set

**Location:** `app/api/cron/daily-quest/route.ts:19-29`, `app/api/cron/streak-reminder/route.ts:19-29`
**Category:** OWASP A01 — Broken Access Control
**CWE:** CWE-306 — Missing Authentication for Critical Function

**Description:**
Both cron endpoints guard themselves only `if (cronSecret)` — making the secret opt-in rather than opt-out. If `CRON_SECRET` is absent from the environment (empty string, or not set at all), the guard block is skipped entirely and the endpoint processes any unauthenticated POST. An attacker who discovers these URLs can trigger fan-out push notifications to all users at will, causing notification spam and consuming VAPID quota. The `lib/env.ts` schema marks `CRON_SECRET` as `emptyToUndefined` and `optional()`, so the app will start happily without it.

In production Kubernetes deployments this is particularly risky: the endpoints are internet-reachable, and there is no network-layer restriction documented in the codebase.

**Recommendation:**
Invert the guard to fail-closed: if `cronSecret` is falsy, return 401 immediately (or 403 with a log warning). Only skip the check explicitly in `NODE_ENV === 'development'`. Change the env schema to require `CRON_SECRET` in production by using a conditional Zod refinement (e.g., `.refine(val => process.env.NODE_ENV !== 'production' || !!val)`).

**References:** CWE-306, OWASP ASVS 4.3.1

### [HIGH] — SEC-03: Race Condition in `completeTask` — Coin Manipulation (TOCTOU)

**Location:** `lib/tasks.ts:215-308`
**Category:** OWASP A04 — Insecure Design
**CWE:** CWE-362 — Race Condition / TOCTOU

**Description:**
`completeTask` performs a multi-step sequence of separate database operations without wrapping them in a transaction:

1. `getTaskById` — read task + check ownership + check `completedAt`
2. `db.update(tasks)` — set `completedAt` or `nextDueDate`
3. `db.insert(taskCompletions)` — record event
4. `incrementUserCoins` — read coins, compute new total, write back
5. `updateStreak`, `checkAndUnlockAchievements`, level update

Between steps 1 and 2, a concurrent second request for the same task can pass the `completedAt === null` check before either request has written `completedAt`. This allows a user to complete the same task multiple times in rapid succession, receiving duplicate coin awards. The rate limiter (30 completions/minute) reduces the window but does not eliminate it, and the rate limiter is per-instance (see SEC-10).

The coin update in `incrementUserCoins` itself has a read-modify-write pattern (steps: SELECT coins, compute, UPDATE) that is also vulnerable to concurrent coin inflation.

**Recommendation:**
Wrap the entire `completeTask` body in a Drizzle/pg transaction (`db.transaction(async (tx) => { ... })`). Use `SELECT ... FOR UPDATE` or optimistic locking on the task row to serialize concurrent completions. At minimum, add a database-level check: use `UPDATE tasks SET completed_at = now() WHERE id = $1 AND user_id = $2 AND completed_at IS NULL RETURNING *` and treat 0 rows returned as "already completed" — this makes the check-and-set atomic.

**References:** CWE-362, OWASP ASVS 11.1.6

### [HIGH] — SEC-04: Internal Error Messages Forwarded to Client

**Location:** Multiple routes — see full list below
**Category:** OWASP A05 — Security Misconfiguration / Information Disclosure
**CWE:** CWE-209 — Generation of Error Message Containing Sensitive Information

**Description:**
Twelve API route handlers use the pattern `error instanceof Error ? error.message : "<fallback>"` and return `error.message` directly in the HTTP response body. This propagates raw Drizzle ORM error messages, PostgreSQL driver errors, and Node.js runtime errors to the caller.

Affected files:
- `app/api/tasks/[id]/route.ts` — PATCH (line 84), DELETE (line 110)
- `app/api/tasks/[id]/complete/route.ts` — POST (line 54), DELETE (line 85)
- `app/api/daily-quest/postpone/route.ts` — POST (line 54)
- `app/api/wishlist/[id]/route.ts` — PATCH (line 56), DELETE (line 83)
- `app/api/wishlist/[id]/buy/route.ts` — POST (line 36), DELETE (line 63)
- `app/api/wishlist/[id]/discard/route.ts` — POST (line 32)
- `app/api/topics/[id]/route.ts` — PATCH (line 85), DELETE (line 112)

Some of these error messages are intentionally informational (e.g., `"Task not found or access denied"`, `"Task is already completed"`, `"Recurring tasks cannot be uncompleted"`), which is acceptable. The risk is that unexpected database or library errors — which contain table names, column names, constraint names, and connection details — will leak to the client unchanged.

A PostgreSQL unique constraint violation, for instance, produces messages like: `duplicate key value violates unique constraint "user_achievements_pkey"` — revealing the schema structure.

**Recommendation:**
Introduce a two-tier error classification. Errors thrown by business logic functions (defined in `lib/`) that carry "safe" messages for the user should implement a marker interface (e.g., `class AppError extends Error { readonly isAppError = true }`). API routes should pass `AppError.message` through, but for all other errors log the full error server-side and return a generic `"An unexpected error occurred"` message with a correlation ID. This pattern is consistent with the existing CLAUDE.md rule: "never expose stack traces to the client."

**References:** CWE-209, OWASP ASVS 8.3.4

### [HIGH] — SEC-05: No UNIQUE Constraint on `user_achievements` — Duplicate Achievement Exploit

**Location:** `lib/db/schema.ts:322-334`
**Category:** OWASP A04 — Insecure Design
**CWE:** CWE-284 — Improper Access Control

**Description:**
The `user_achievements` table has no composite UNIQUE constraint on `(user_id, achievement_id)`. The application-layer guard in `checkAndUnlockAchievements` queries for existing rows and filters them out, but this check is not atomic. Under the race condition in SEC-03, or if `checkAndUnlockAchievements` is called concurrently, duplicate rows can be inserted. This does not grant extra coins (achievements and coins are separate), but it breaks the achievement system integrity — a user could appear to have earned the same achievement multiple times, and the deduplication logic would need to be handled at query time everywhere.

**Recommendation:**
Add a composite unique constraint to the schema:

```typescript
export const userAchievements = pgTable(
  "user_achievements",
  { ... },
  (table) => ({
    uniqueUserAchievement: unique().on(table.userId, table.achievementId),
  })
);
```

Then in `checkAndUnlockAchievements`, use `.onConflictDoNothing()` on the insert (consistent with `seedAchievements`) rather than the two-step select-then-insert pattern.

**References:** CWE-284, OWASP ASVS 6.2.7

### [MEDIUM] — SEC-07: Rate Limiting Is In-Memory — Bypassable in Multi-Replica Deployments

**Location:** `lib/rate-limit.ts:1-59`
**Category:** OWASP A04 — Insecure Design
**CWE:** CWE-770 — Allocation of Resources Without Limits

**Description:**
The rate limiter uses a module-level `Map` as its backing store. Each Node.js process has its own independent map. In a multi-replica deployment (Kubernetes with `replicas > 1`, or any horizontal scaling), the effective rate limit for a user is multiplied by the number of pods. With 3 replicas, a user can make 3x the configured limit before any single replica triggers a 429 response. The code already documents this limitation in its JSDoc comment, but the limitation is not enforced or mitigated anywhere.

Additionally, the `store` Map is never pruned of expired entries, causing unbounded memory growth proportional to the number of unique rate limit keys seen since process start. For a long-running server with many users, this constitutes a memory leak.

**Recommendation:**
- For production multi-replica deployments, replace the in-memory store with a Redis-based sliding window implementation (e.g., using `ioredis` with a Lua script, or the `rate-limiter-flexible` library).
- Add a periodic cleanup of expired entries: iterate the store and delete entries where `now >= entry.resetAt`. This can be done lazily on each `checkRateLimit` call (evict expired entries for the accessed key) or via a `setInterval` cleanup task.

**References:** CWE-770, OWASP ASVS 13.4.1

### [MEDIUM] — SEC-09: Timing Attack on CRON_SECRET Comparison

**Location:** `app/api/cron/daily-quest/route.ts:26`, `app/api/cron/streak-reminder/route.ts:27`
**Category:** OWASP A02 — Cryptographic Failures
**CWE:** CWE-208 — Observable Timing Discrepancy

**Description:**
The cron secret is compared using JavaScript's `!==` operator (`if (token !== cronSecret)`). String comparison in JavaScript is not constant-time — it short-circuits at the first non-matching character. An attacker with the ability to make many requests and measure response times (or use statistical timing analysis over a network) could theoretically brute-force the secret one character at a time. While this attack is harder to execute over the internet due to network jitter, it is best practice to use constant-time comparison for secret tokens.

**Recommendation:**
Use Node.js's `crypto.timingSafeEqual` for the comparison:

```typescript
import { timingSafeEqual } from "crypto";

const tokenBuffer = Buffer.from(token ?? "", "utf8");
const secretBuffer = Buffer.from(cronSecret, "utf8");
if (tokenBuffer.length !== secretBuffer.length || !timingSafeEqual(tokenBuffer, secretBuffer)) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

**References:** CWE-208, OWASP ASVS 2.9.1

### [MEDIUM] — SEC-10: `NEXT_PUBLIC_VAPID_PUBLIC_KEY` Accessed Directly in `lib/push.ts`

**Location:** `lib/push.ts:65`, `lib/push.ts:79`
**Category:** OWASP A05 — Security Misconfiguration
**CWE:** CWE-330 — Use of Insufficiently Random Values (misconfiguration pathway)

**Description:**
`lib/push.ts` accesses `process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY` directly rather than through the validated `clientEnv` wrapper in `lib/env.ts`. This bypasses the Zod validation layer and the project's own architecture rule (CLAUDE.md: "All env vars accessed via a typed wrapper in `lib/env.ts`"). The consequence is:

1. If the env var is missing or malformed, `isVapidConfigured()` silently returns `false` and all push notifications are silently dropped — no startup error, no clear diagnostic.
2. The value is used with a non-null assertion (`process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!`) on line 79, which will pass `undefined` to `webpush.setVapidDetails()` if the variable is absent, potentially causing a runtime error with an opaque message from the `web-push` library.

This was also identified in the L1 peer review (NIT-02) and L2 architecture review but remains unaddressed.

**Recommendation:**
Replace both direct `process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY` accesses with `clientEnv.NEXT_PUBLIC_VAPID_PUBLIC_KEY`. This ensures the validation layer catches misconfiguration at startup.

**References:** CLAUDE.md architecture rule, L1-peer-20260331 NIT-02

### [MEDIUM] — SEC-11: OAuth Tokens Stored in Plain Text in Database

**Location:** `lib/db/schema.ts:138-143`
**Category:** OWASP A02 — Cryptographic Failures
**CWE:** CWE-311 — Missing Encryption of Sensitive Data

**Description:**
The `accounts` table stores `refresh_token`, `access_token`, and `id_token` as plain `text` columns. These are OAuth tokens granted by GitHub, Discord, and Google. While these are required columns for the Auth.js Drizzle adapter schema, storing them unencrypted means that:
- A database dump or SQL injection vulnerability exposes long-lived OAuth tokens directly.
- An attacker with read access to the database can impersonate users at their OAuth providers.

**Recommendation:**
Auth.js does not natively encrypt adapter columns. Options:
1. Use column-level encryption: encrypt/decrypt tokens in the adapter hooks using `AES-256-GCM` with a key derived from `AUTH_SECRET`. This can be done transparently via Drizzle's custom type mappings.
2. Accept the risk if the application's threat model treats database compromise as fully out-of-scope (i.e., the database is not separately accessible).
3. At minimum, ensure PostgreSQL connection uses TLS (`ssl: true` in the pg client config) and the database is not internet-facing.

**References:** CWE-311, OWASP ASVS 6.2.1

### [MEDIUM] — SEC-12: CSP Allows `unsafe-eval` and `unsafe-inline` for `script-src` in Production

**Location:** `next.config.ts:59`
**Category:** OWASP A05 — Security Misconfiguration
**CWE:** CWE-116 — Improper Encoding or Escaping of Output

**Description:**
The Content Security Policy includes `script-src 'self' 'unsafe-eval' 'unsafe-inline'`. Both directives are applied in production (not gated on `NODE_ENV`). This effectively neutralizes XSS mitigations provided by CSP:
- `'unsafe-eval'` allows `eval()`, `new Function()`, and similar dynamic code execution — the primary mechanism exploited by XSS payloads.
- `'unsafe-inline'` allows inline `<script>` tags and event handler attributes.

The `frame-ancestors 'none'` directive correctly prevents clickjacking, but the CSP's script-src restrictions do not provide meaningful XSS protection in this configuration. The comment in the config acknowledges this and suggests nonces.

**Recommendation:**
Implement nonce-based CSP for production. Next.js 15 supports CSP nonces via middleware. This allows removal of `'unsafe-eval'` and `'unsafe-inline'` while maintaining compatibility with Next.js hydration. See the [Next.js CSP with nonces documentation](https://nextjs.org/docs/app/building-your-application/configuring/content-security-policy). This is a multi-step change that requires middleware configuration and may require adjustments to third-party scripts.

**References:** CWE-116, OWASP ASVS 14.4.1

### [MEDIUM] — SEC-13: Push Subscription Endpoint Stored Without Endpoint Domain Validation

**Location:** `app/api/push/subscribe/route.ts:22-27`
**Category:** OWASP A03 — Injection (SSRF pathway)
**CWE:** CWE-918 — Server-Side Request Forgery

**Description:**
The `pushSubscriptionSchema` validates that `endpoint` is a URL (`z.string().url()`) but does not restrict which domains are permitted. A malicious authenticated user could submit a push subscription with an endpoint pointing to an internal service (e.g., `http://localhost:5432/`, `http://169.254.169.254/` for cloud metadata endpoints, or `http://internal-service/`). When the server calls `webpush.sendNotification(subscription, ...)`, the `web-push` library makes an outbound HTTP POST to the provided endpoint URL. This is a server-side request forgery vector.

The impact is limited by the fact that `web-push` sends a specific WebPush protocol payload with specific headers, but the outbound request itself reaches the target, which may be sufficient to trigger internal services that act on POST requests.

**Recommendation:**
Add an allowlist of permitted push service domains to the Zod schema. Standard push services are:
- `https://fcm.googleapis.com/`
- `https://*.push.services.mozilla.com/`
- `https://*.notify.windows.com/`
- `https://*.pushapi.apple.com/`

Reject endpoints not matching these patterns. Example:

```typescript
endpoint: z.string().url().refine(
  (url) => ALLOWED_PUSH_ORIGINS.some(origin => url.startsWith(origin)),
  { message: "Endpoint must be a recognized push service" }
),
```

**References:** CWE-918, OWASP ASVS 12.6.1

### [LOW] — SEC-14: `checkCompletedToday` Issues Two Sequential Queries Instead of One Filtered Query

**Location:** `lib/push.ts:262-334`
**Category:** OWASP A04 — Insecure Design (logic error with correctness impact)
**CWE:** CWE-1041 — Use of Redundant Code

**Description:**
In `sendStreakReminders`, the code first fetches completions with `isNotNull(completedAt)` into `completionsToday` (line 262), then conditionally calls `checkCompletedToday` if rows exist (line 276). `checkCompletedToday` itself runs another full query of all completions for the user and filters in JavaScript using `new Date(r.completedAt) >= todayStart`.

The intermediate `completionsToday` query at line 262 serves no purpose: the entire logic is replicated in `checkCompletedToday`. The `todayStart` variable computed on line 237 is not used in the first query and only reaches `checkCompletedToday` because it is passed as a parameter. More critically, neither query applies the actual date filter at the database level — they fetch all completions for the user across all time and filter in JavaScript. For users with many historical completions, this is a performance problem, but it is also a functional issue: it relies on JavaScript date comparison rather than the database, introducing timezone sensitivity.

This is a logic defect that also represents a potential correctness gap in the security-relevant "has user completed a task today" check.

**Recommendation:**
Replace both queries with a single database-level query using a `gte` (greater than or equal) filter on `completedAt`. Drizzle ORM supports date comparisons with `gte(taskCompletions.completedAt, todayStart)`.

**References:** CWE-1041

### [LOW] — SEC-15: Auth.js CSRF Protection Scope

**Location:** `lib/auth.ts`, `app/api/auth/[...nextauth]/route.ts`
**Category:** OWASP A01 — Broken Access Control (CSRF)
**CWE:** CWE-352 — Cross-Site Request Forgery

**Description:**
Auth.js v5 (next-auth 5.x) provides built-in CSRF protection for its own sign-in/sign-out endpoints. Database session strategy is used (`strategy: "database"`), which means session tokens are stored in the database and validated per-request. Auth.js generates a CSRF token for its own form submissions.

However, the application's own API routes (`/api/tasks/...`, `/api/wishlist/...`, etc.) rely solely on `auth()` — the session cookie presence check — and do not implement additional CSRF protection. Same-site cookie attributes are not configured in `lib/auth.ts` (Auth.js sets `sameSite: "lax"` by default). With `SameSite=Lax`, cross-site top-level navigations can send cookies, but cross-site XHR/fetch requests cannot. For a Next.js App Router SPA, the API routes are called via `fetch()` from the same origin, so `SameSite=Lax` provides adequate protection against cross-site fetch-based CSRF.

This is LOW because the default Auth.js `SameSite=Lax` cookie behavior is adequate for the current usage pattern. It becomes HIGH if the application ever serves content from a different subdomain or if the cookie attributes are changed.

**Recommendation:**
Explicitly confirm that `SameSite=Lax` or `SameSite=Strict` is set on session cookies (check via browser devtools in production). Consider adding `SameSite=Strict` via the Auth.js `cookies` configuration to provide defense-in-depth. Document the CSRF posture.

**References:** CWE-352, OWASP ASVS 4.2.2

### [LOW] — SEC-16: Rate Limit Store Memory Leak — No Entry Expiration

**Location:** `lib/rate-limit.ts:21`
**Category:** OWASP A04 — Insecure Design
**CWE:** CWE-401 — Missing Release of Memory After Effective Lifetime

**Description:**
The `store` Map in `lib/rate-limit.ts` is never pruned. Each unique rate limit key (e.g., `tasks-create:<userId>`, `tasks-complete:<userId>`) creates a permanent entry in the Map, even after its window expires. Over time, with many users and many rate-limit-keyed endpoints, the Map grows without bound. This is a server memory leak that worsens over long uptime and with a large user base.

**Recommendation:**
Add lazy expiration: when `checkRateLimit` is called, if the found entry has expired (`now >= entry.resetAt`), delete it and create a fresh entry. Also consider adding a periodic sweep via `setInterval` to collect all expired entries (e.g., every 5 minutes).

### [LOW] — SEC-17: `notes` Field on Tasks Has No Length Limit

**Location:** `lib/validators/index.ts:23-57`, `lib/db/schema.ts:221`
**Category:** OWASP A04 — Insecure Design
**CWE:** CWE-400 — Uncontrolled Resource Consumption

**Description:**
The `notes` field in `CreateTaskInputSchema` and `UpdateTaskInputSchema` is validated as `z.string().nullable().optional()` with no maximum length. A user can submit a task note of arbitrary size (limited only by HTTP request body size and PostgreSQL's `text` type, which accepts up to 1 GB). While not immediately exploitable, this allows storage exhaustion attacks: a malicious authenticated user could fill the database with enormous note fields.

**Recommendation:**
Add a reasonable maximum length: `z.string().max(10000, "Notes cannot exceed 10,000 characters").nullable().optional()`. Apply similar limits to `description` in topics and `icon`/`color` in topics.

## Positive Security Findings

The following security controls are correctly implemented and should be preserved:
