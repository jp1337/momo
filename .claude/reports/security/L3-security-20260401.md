### [HIGH] — D-01: Incorrect HTTP Status Code Allows Client to Distinguish "Not Bought" vs "Not Found" States

**Status:** Residual — B-03 from L1 review, not yet fixed
**Location:** `app/api/wishlist/[id]/buy/route.ts:64-70`
**Category:** OWASP A01 — Broken Access Control (information disclosure)
**CWE:** CWE-203 — Observable Discrepancy

**Description:**
The `DELETE /api/wishlist/:id/buy` handler catches two semantically different errors from `unmarkAsBought()` in `lib/wishlist.ts` and maps them both to HTTP 404:

- `"Wishlist item not found or access denied"` — item does not exist or belongs to another user
- `"Wishlist item is not marked as bought"` — item exists and is owned by caller, but has wrong status

The predicate `isNotFound` matches on both `"not found"` and `"not marked"` and returns 404 in both cases. The response body text in both cases is `"Item not found"`.

While the response body is sanitized (no raw `error.message` is returned), the HTTP status code itself leaks state. An attacker enumerating UUIDs can distinguish:
- 401: not authenticated
- 404 (after any auth): item not found OR wrong status — conflated, which is correct behavior for the auth-bypass case

On its own this conflation is semantically acceptable for the not-found vs. access-denied ambiguity (standard practice), but the conflation with a business-logic state error (`"not marked as bought"`) is incorrect and creates a subtle contract problem.

The correct semantic is:
- Item not found / not owned → 404
- Item found and owned, but not in BOUGHT state → **409 Conflict**

This distinction matters because:
1. A 409 confirms the item exists and is owned by the caller (mild information disclosure).
2. A 404 for a business-logic state error hides real application errors from clients, making debugging and integration harder.
3. If the client retries on 404 (reasonable), it will loop indefinitely on a valid-but-wrong-state item.

**Recommendation:**
Split the `isNotFound` predicate into two distinct cases. Return 409 when `error.message.includes("not marked")` and 404 only when `error.message.includes("not found")`. This matches the pattern already used correctly in `app/api/tasks/[id]/complete/route.ts:87-95` (which correctly returns 409 for `"not completed"`).

**References:** CWE-203, RFC 9110 §15.5.10

### [MEDIUM] — D-02: No Top-Level `permissions` Block in `build-and-publish.yml` — Lint Job Has Implicit Write Permissions

**Status:** New finding
**Location:** `.github/workflows/build-and-publish.yml` (entire file, no top-level `permissions` key)
**Category:** OWASP A05 — Security Misconfiguration
**CWE:** CWE-250 — Execution with Unnecessary Privileges

**Description:**
The `build-and-publish.yml` workflow has no top-level `permissions` declaration. GitHub Actions resolves permissions as follows when no top-level block is present:

- The `build` and `merge` jobs each declare `permissions: { contents: read, packages: write }` — these are correctly scoped.
- The `lint` job declares **no** `permissions` block at all.

When a job has no `permissions` block and the workflow has no top-level `permissions` block, the job inherits the **repository default token permissions**. For public repositories this defaults to `read-all` in most GitHub organization configurations. However, if the repository owner's organization (or account) has configured the default to `read-and-write`, the `lint` job's `GITHUB_TOKEN` will have write access to all repository scopes including `contents: write`, `issues: write`, `pull-requests: write`, and `actions: write`.

The lint job runs `npm ci` which executes arbitrary code from `node_modules/` (including lifecycle scripts). A compromised dependency in the supply chain could exfiltrate the high-permission token.

**Recommendation:**
Add a top-level `permissions` block that denies everything by default:

```yaml
permissions: {}
```

Then keep the explicit per-job grants on `build` and `merge`. Add an explicit `permissions: {}` block to the `lint` job as well (it needs no token access). This follows the GitHub-recommended principle of least privilege for workflow tokens.

**References:** CWE-250, GitHub Docs — Workflow permissions, SLSA Build L3

### [MEDIUM] — D-03: `docs.yml` Uses Non-Pinned Mutable Action Tags — Supply Chain Risk

**Status:** New finding
**Location:** `.github/workflows/docs.yml:24, 27, 42, 46, 49`
**Category:** OWASP A06 — Vulnerable and Outdated Components
**CWE:** CWE-494 — Download of Code Without Integrity Check

**Description:**
The `docs.yml` workflow references GitHub Actions by mutable version tags rather than immutable commit SHAs:

| Step | Action | Tag Used | Current Latest |
|------|--------|----------|----------------|
| Checkout | `actions/checkout` | `@v6` | `v6.0.2` |
| Configure Pages | `actions/configure-pages` | `@v6` | `v6.0.0` |
| Upload artifact | `actions/upload-pages-artifact` | `@v4` | `v4.0.0` |
| Deploy Pages | `actions/deploy-pages` | `v5` | `v5.0.0` |

Mutable tags (`@v6`, `@v5`) can be silently redirected by a compromised maintainer or GitHub account to point to different, malicious commits. Because this workflow has `id-token: write` and `pages: write` permissions, a compromised action could:
1. Exfiltrate the OIDC token (usable to publish arbitrary content to GitHub Pages)
2. Deploy attacker-controlled content to the project's public documentation site

Note: `build-and-publish.yml` already uses mutable tags (`@v3`, `@v4`, `@v6`) across its steps, amplifying the overall supply chain exposure — but this is out of scope for this delta review.

**Recommendation:**
Pin all action references to their full commit SHA. Use a tool such as `pin-github-action` or `Dependabot` with `update-type: security-patch` to maintain pinned SHAs automatically. Example:

```yaml
# Before
uses: actions/checkout@v6
# After
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v6.0.2
```

**References:** CWE-494, SLSA Build L2, GitHub Docs — Hardening security for GitHub Actions

### [MEDIUM] — D-04: `lodash` Override Does Not Remediate GHSA-xxjr-mmjv-4gpg — No Fix Exists in 4.x

**Status:** Residual — npm override partially effective
**Location:** `package.json:29`
**Category:** OWASP A06 — Vulnerable and Outdated Components
**CWE:** CWE-1321 — Prototype Pollution

**Description:**
The `package.json` override `"lodash": "4.17.21"` was introduced to avoid the broken `4.18.0` release (deprecated by the lodash maintainer with the message "Bad release. Please use lodash@4.17.21 instead."). This is correct for the broken-release concern.

However, `npm audit` confirms that `4.17.21` remains within the vulnerable range for **GHSA-xxjr-mmjv-4gpg** (Prototype Pollution in `_.unset` and `_.omit`, CVSS 6.5, CWE-1321):

```
Vulnerable range: >=4.0.0 <=4.17.22
Resolved version: 4.17.21 overridden
```

There is no patched version of lodash 4.x available on the `latest` dist-tag. Version `4.17.23` exists in the registry but is not tagged `latest` and is not confirmed to contain a fix for this advisory.

**Mitigating factors:**
- Lodash is used only by `workbox-build` and `workbox-webpack-plugin`, both of which are transitive dependencies of `@ducanh2912/next-pwa` (a devDependency used only at build time).
- Lodash is **not present** in the production runtime bundle — it is a build-tool dependency only.
- The prototype pollution vector (`_.unset`, `_.omit`) requires the attacker to control the input object passed to lodash functions during the build process, which is not a realistic attack surface in a CI/CD build.

**Risk assessment:** The practical exploitability is low given the build-only context, but the advisory remains open and `npm audit` will continue to report it.

**Recommendation:**
Accept the residual risk with a documented exception in this report. Monitor the lodash advisory for a 4.x fix. Alternatively, evaluate whether `@ducanh2912/next-pwa` can be updated to a version that drops the lodash dependency — the `npm audit` output suggests `10.2.6` (isSemVerMajor: true) may resolve it, though the "isSemVerMajor" flag here may be misleading (it is within the same `10.x` line and worth testing). Add a comment to `package.json` near the override explaining the reason and accepted risk.

**References:** GHSA-xxjr-mmjv-4gpg, CWE-1321

### [LOW] — D-05: SEC-04 Fix Is Incomplete for `DELETE /api/wishlist/:id/buy` — Misclassification of "Not Marked" Error

**Status:** Partially fixed — SEC-04 from baseline
**Location:** `app/api/wishlist/[id]/buy/route.ts:64-70`
**Category:** OWASP A05 — Security Misconfiguration (error handling)
**CWE:** CWE-209 — Generation of Error Message Containing Sensitive Information

**Description:**
The SEC-04 fix correctly prevents raw `error.message` content from being forwarded to the client. All three changed routes now return only hardcoded strings (`"Item not found"` or `"Internal server error"`). This is correct and closes the information disclosure path for both `POST /api/wishlist/:id/buy` and `POST /api/wishlist/:id/discard`.

However, for `DELETE /api/wishlist/:id/buy`, the predicate used to classify the error is:

```typescript
const isNotFound =
  error instanceof Error &&
  (error.message.includes("not found") || error.message.includes("not marked"));
```

The `"not marked"` branch matches `"Wishlist item is not marked as bought"` thrown by `unmarkAsBought()` when the item exists but has the wrong status. This error is classified as "not found" and returns HTTP 404 with body `"Item not found"`. This is a semantic incorrectness (covered in D-01 above as the higher-severity finding), but also represents an incomplete fix: the fix introduced a new string match (`"not marked"`) that was not present in the original route and extended the classification rather than correcting it.

**Recommendation:**
The `"not marked"` condition should be separated and map to a 409 response, not included in the `isNotFound` predicate. See D-01 for the full recommendation. This finding is marked Low here because the information disclosure aspect of SEC-04 is correctly resolved; the remaining issue is a correctness and semantic problem.

**References:** CWE-209

### [LOW] — D-06: `lib/auth.ts` Static Import Change — No Security Regression, Minor Improvement

**Status:** Verification — confirmed correct
**Location:** `lib/auth.ts:18`
**Category:** OWASP A06 — Vulnerable and Outdated Components
**CWE:** N/A

**Description:**
The Keycloak provider was changed from a dynamic `require()` call to a static ES module import (`import Keycloak from "next-auth/providers/keycloak"`). This is a security improvement:

- Dynamic `require()` bypasses TypeScript's type system and module resolution checks, potentially allowing runtime substitution.
- Static imports are resolved at compile time, giving the TypeScript compiler and bundler visibility into the import graph.
- The module is conditional on `serverEnv.OIDC_ISSUER` being set (line 57), which correctly uses the validated `serverEnv` wrapper rather than raw `process.env`.

No security regression is introduced. The change is verified correct.

**References:** N/A — informational only
